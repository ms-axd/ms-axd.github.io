이번에는 `EIP-1155`를 보자.

EIP-1155는 흔히 `ERC-1155`라고 부르는 `Multi Token Standard`다.
ERC-20과 ERC-721을 따로따로 쓰는 구조에서 생기는 비효율을 줄이기 위해 등장했다.

```text
ERC-20  -> 대체 가능 토큰
ERC-721 -> 대체 불가능 토큰
ERC-1155 -> 둘 다 한 컨트랙트에서 다루는 멀티 토큰 표준
```

---

## EIP-1155란?

`EIP-1155`는 하나의 스마트 컨트랙트에서 여러 종류의 토큰을 관리할 수 있게 만든 표준이다.

ERC-20은 한 컨트랙트가 보통 하나의 토큰을 관리한다.
ERC-721도 한 컨트랙트가 보통 하나의 NFT 컬렉션을 관리한다.

반면 ERC-1155는 하나의 컨트랙트 안에서 여러 `id`를 사용해 다양한 토큰을 표현한다.

```text
id 1 -> 골드 코인
id 2 -> 나무
id 3 -> 전설 검
id 4 -> 이벤트 티켓
```

각 `id`는 토큰 종류를 의미한다.
그리고 각 `id`별로 수량을 가질 수 있다.

즉 ERC-1155는 ERC-20처럼 수량을 다룰 수도 있고, ERC-721처럼 고유 자산처럼 사용할 수도 있다.

---

## 왜 등장했나

ERC-20과 ERC-721만으로도 토큰을 만들 수는 있다.
하지만 여러 종류의 자산을 많이 다뤄야 하는 서비스에서는 비효율이 생긴다.

대표적인 예가 게임이다.

게임에는 다음과 같은 자산이 함께 존재한다.

```text
골드 1000개
포션 20개
나무 50개
전설 검 1개
캐릭터 스킨 1개
이벤트 티켓 3개
```

이걸 ERC-20과 ERC-721로만 구현하면 자산 종류마다 컨트랙트를 따로 만들거나, 복잡한 구조를 직접 설계해야 한다.

```text
GoldToken.sol
PotionToken.sol
WoodToken.sol
LegendSwordNFT.sol
SkinNFT.sol
TicketNFT.sol
```

자산 종류가 적으면 괜찮다.
하지만 아이템이 수백 개, 수천 개가 되면 배포 비용, 관리 비용, 전송 비용이 커진다.

EIP-1155는 이 문제에서 출발했다.

```text
여러 종류의 토큰을 하나의 컨트랙트에서 표준 방식으로 관리하자.
```

---

## EIP-1155가 해결한 문제

### 1. 컨트랙트 수 증가 문제

ERC-20과 ERC-721은 토큰 종류가 많아질수록 컨트랙트도 늘어나기 쉽다.

컨트랙트가 많아지면 다음 문제가 생긴다.

```text
배포 비용 증가
관리 복잡도 증가
권한 관리 지점 증가
마켓플레이스 연동 비용 증가
인덱싱 비용 증가
```

ERC-1155는 여러 토큰을 하나의 컨트랙트에서 관리해 이 문제를 줄인다.

### 2. 대량 전송 문제

ERC-721에서 NFT 여러 개를 보내려면 보통 전송을 여러 번 호출해야 한다.
ERC-20도 여러 종류의 토큰을 보내려면 각 컨트랙트를 따로 호출해야 한다.

ERC-1155는 `safeBatchTransferFrom`을 통해 여러 토큰을 한 번에 전송할 수 있다.

```text
id 1, amount 100
id 2, amount 50
id 3, amount 1
```

이런 묶음을 한 번의 호출로 보낼 수 있다.
게임 아이템 지급, 에어드랍, 번들 판매 같은 상황에서 유리하다.

### 3. 대체 가능/대체 불가능 토큰 분리 문제

서비스 안에는 대체 가능한 자산과 대체 불가능한 자산이 함께 존재한다.

```text
골드, 포션, 재료 -> 대체 가능
전설 검, 캐릭터, 랜드 -> 대체 불가능
```

ERC-1155는 둘을 하나의 모델로 다룬다.

대체 가능한 토큰은 같은 `id`에 여러 수량을 둔다.
대체 불가능한 토큰은 특정 `id`의 수량을 1개만 발행하는 방식으로 표현할 수 있다.

## ERC-1155 핵심 구조

ERC-1155의 핵심은 `id`와 `amount`다.

| 개념 | 의미 |
| --- | --- |
| `id` | 토큰 종류 |
| `amount` | 해당 토큰의 수량 |
| `account` | 토큰을 보유한 주소 |

ERC-20은 주소별 잔액을 본다.
ERC-721은 tokenId별 소유자를 본다.
ERC-1155는 주소와 id의 조합으로 잔액을 본다.

```text
balanceOf(alice, 1) -> Alice가 id 1 토큰을 몇 개 가지고 있는가
balanceOf(alice, 2) -> Alice가 id 2 토큰을 몇 개 가지고 있는가
```

이 구조 덕분에 하나의 컨트랙트에서 여러 자산을 관리할 수 있다.

---

## ERC-1155 핵심 함수

ERC-1155의 핵심 함수는 다음과 같다.

| 함수 | 설명 |
| --- | --- |
| `balanceOf(address,uint256)` | 특정 주소가 특정 id 토큰을 얼마나 가지고 있는지 조회 |
| `balanceOfBatch(address[],uint256[])` | 여러 주소/id 조합의 잔액을 한 번에 조회 |
| `setApprovalForAll(address,bool)` | operator에게 전체 토큰 관리 권한 부여 |
| `isApprovedForAll(address,address)` | operator 승인 여부 조회 |
| `safeTransferFrom(address,address,uint256,uint256,bytes)` | 특정 id 토큰 전송 |
| `safeBatchTransferFrom(address,address,uint256[],uint256[],bytes)` | 여러 id 토큰 일괄 전송 |

대략적인 인터페이스는 다음과 같다.

```solidity
interface IERC1155 {
    function balanceOf(address account, uint256 id) external view returns (uint256);

    function balanceOfBatch(
        address[] calldata accounts,
        uint256[] calldata ids
    ) external view returns (uint256[] memory);

    function setApprovalForAll(address operator, bool approved) external;
    function isApprovedForAll(address account, address operator) external view returns (bool);

    function safeTransferFrom(
        address from,
        address to,
        uint256 id,
        uint256 amount,
        bytes calldata data
    ) external;

    function safeBatchTransferFrom(
        address from,
        address to,
        uint256[] calldata ids,
        uint256[] calldata amounts,
        bytes calldata data
    ) external;
}
```

---

## 주요 함수 분석

### balanceOf

`balanceOf`는 특정 주소가 특정 id 토큰을 얼마나 보유했는지 확인한다.

```solidity
balanceOf(alice, 1);
```

이 호출은 Alice가 `id 1` 토큰을 몇 개 가지고 있는지 반환한다.

ERC-20의 `balanceOf(address)`와 비슷하지만, ERC-1155는 `id`가 추가된다.

### balanceOfBatch

`balanceOfBatch`는 여러 잔액을 한 번에 조회한다.

```solidity
balanceOfBatch(
    [alice, alice, bob],
    [1, 2, 1]
);
```

이런 호출은 다음 질문을 한 번에 처리한다.

```text
Alice가 id 1을 몇 개 가지고 있는가
Alice가 id 2를 몇 개 가지고 있는가
Bob이 id 1을 몇 개 가지고 있는가
```

여러 아이템 잔액을 화면에 표시해야 하는 게임이나 지갑에서 유용하다.

### safeTransferFrom

`safeTransferFrom`은 특정 id의 특정 수량을 전송한다.

```solidity
safeTransferFrom(alice, bob, 1, 100, "");
```

이 호출은 Alice가 가진 `id 1` 토큰 100개를 Bob에게 보낸다.

ERC-721에서는 `tokenId` 하나를 보내지만, ERC-1155에서는 `id`와 `amount`를 같이 보낸다.

### safeBatchTransferFrom

`safeBatchTransferFrom`은 ERC-1155의 핵심 장점이다.

```solidity
safeBatchTransferFrom(
    alice,
    bob,
    [1, 2, 3],
    [100, 50, 1],
    ""
);
```

이 호출은 다음 전송을 한 번에 처리한다.

```text
id 1 -> 100개
id 2 -> 50개
id 3 -> 1개
```

여러 토큰을 한 번에 옮길 수 있어서 전송 비용과 호출 복잡도를 줄일 수 있다.

### setApprovalForAll

ERC-1155도 ERC-721처럼 operator 승인 구조를 사용한다.

```solidity
setApprovalForAll(marketplace, true);
```

이 권한을 주면 마켓플레이스가 사용자의 ERC-1155 토큰을 대신 전송할 수 있다.

편하지만 위험하다.
악성 컨트랙트에 승인하면 해당 ERC-1155 컨트랙트 안의 여러 자산이 한꺼번에 위험해질 수 있다.

---

## 이벤트

ERC-1155는 단일 전송과 배치 전송을 구분해서 이벤트를 발생시킨다.

| 이벤트 | 의미 |
| --- | --- |
| `TransferSingle` | 하나의 id 전송 |
| `TransferBatch` | 여러 id 일괄 전송 |
| `ApprovalForAll` | operator 승인 |
| `URI` | 특정 id의 메타데이터 URI 변경 |

인덱서와 마켓플레이스는 이 이벤트를 추적해 소유권과 잔액 변화를 계산한다.

특히 ERC-1155는 `TransferBatch`가 중요하다.
여러 토큰 변화가 하나의 이벤트로 기록될 수 있기 때문이다.

## 실제 사용처

### 게임 아이템

ERC-1155는 게임 아이템에 잘 맞는다.

게임에는 대체 가능한 아이템과 대체 불가능한 아이템이 섞여 있다.

```text
골드 -> 대체 가능
포션 -> 대체 가능
재료 -> 대체 가능
전설 무기 -> 대체 불가능하게 사용 가능
캐릭터 스킨 -> 대체 불가능하게 사용 가능
```

ERC-1155는 이 자산들을 하나의 컨트랙트에서 처리할 수 있다.

### NFT 컬렉션

ERC-721은 각 NFT가 완전히 고유한 컬렉션에 적합하다.
반면 ERC-1155는 같은 NFT를 여러 개 발행하는 구조에 적합하다.

예를 들어 에디션 NFT를 생각해보자.

```text
작품 A: 100개 발행
작품 B: 50개 발행
작품 C: 1개 발행
```

이런 구조는 ERC-1155가 자연스럽다.

### 티켓과 쿠폰

이벤트 티켓이나 쿠폰처럼 같은 종류가 여러 장 존재하는 자산도 ERC-1155로 표현하기 좋다.

```text
VIP 티켓 id 1 -> 100장
일반 티켓 id 2 -> 1000장
할인 쿠폰 id 3 -> 5000장
```

각 티켓 종류를 하나의 id로 두고 수량을 관리하면 된다.

### 번들 거래

여러 아이템을 묶어서 전송하거나 판매하는 구조에도 맞다.

예를 들어 게임 스타터 팩을 생각하면 된다.

```text
골드 1000개
포션 10개
초급 무기 1개
스킨 1개
```

ERC-1155는 이런 묶음을 한 번에 전송할 수 있다.

## ERC-20, ERC-721, ERC-1155 비교

| 항목 | ERC-20 | ERC-721 | ERC-1155 |
| --- | --- | --- | --- |
| 토큰 성격 | 대체 가능 | 대체 불가능 | 둘 다 가능 |
| 기본 단위 | amount | tokenId | id + amount |
| 컨트랙트 구조 | 보통 한 토큰당 한 컨트랙트 | 보통 한 컬렉션당 한 컨트랙트 | 여러 토큰을 한 컨트랙트에서 관리 |
| 대량 전송 | 표준 핵심 아님 | 비효율적 | batch transfer 지원 |
| 대표 사용처 | 코인, 포인트, 거버넌스 토큰 | NFT, 고유 자산 | 게임 아이템, 에디션 NFT, 티켓, 쿠폰 |

핵심 차이는 ERC-1155가 `id`와 `amount`를 동시에 사용한다는 점이다.

ERC-20은 수량만 중요하다.
ERC-721은 고유한 tokenId가 중요하다.
ERC-1155는 토큰 종류인 id와 수량인 amount를 같이 본다.

## EIP-1155의 한계

### 복잡한 인덱싱

ERC-1155는 하나의 컨트랙트에서 여러 토큰을 다루기 때문에 인덱싱이 더 복잡할 수 있다.

ERC-721은 `tokenId -> owner` 구조가 직관적이다.
ERC-1155는 `account + id -> balance` 구조를 추적해야 한다.

특히 `TransferBatch` 이벤트를 제대로 처리하지 않으면 잔액 계산이 틀어질 수 있다.

### NFT 고유성 표현의 애매함

ERC-1155도 수량을 1로 두면 NFT처럼 쓸 수 있다.
하지만 ERC-721처럼 각 tokenId가 자연스럽게 고유 자산이라는 느낌은 약하다.

완전히 고유한 1:1 NFT 컬렉션이라면 ERC-721이 더 직관적일 수 있다.

### 승인 권한 위험

ERC-1155의 `setApprovalForAll`은 강력하다.
한 번 승인하면 해당 컨트랙트의 여러 id 토큰이 전부 operator에 의해 이동될 수 있다.

게임 아이템, 티켓, NFT가 한 컨트랙트에 들어 있다면 피해 범위도 커질 수 있다.

### 메타데이터 관리

ERC-1155는 id별 URI를 통해 메타데이터를 제공한다.
하지만 메타데이터가 외부 서버에 있으면 ERC-721과 마찬가지로 링크 변경, 서버 종료, 중앙화 문제가 생길 수 있다.

---

## EIP-1155의 의미

EIP-1155의 핵심은 토큰 종류가 많아지는 상황을 표준으로 처리했다는 점이다.

ERC-20과 ERC-721이 각각 하나의 토큰 모델에 집중했다면, ERC-1155는 여러 자산이 섞여 있는 실제 서비스 구조에 더 가깝다.

분석 관점에서 보면 다음 의미가 있다.

| 관점 | 의미 |
| --- | --- |
| 기술적 관점 | id + amount 모델로 여러 토큰을 하나의 컨트랙트에서 관리 |
| 생태계 관점 | 게임, 티켓, 에디션 NFT 같은 다중 자산 서비스에 적합 |
| 비용 관점 | batch transfer로 여러 전송을 효율화 |
| 보안 관점 | setApprovalForAll 권한의 피해 범위가 커질 수 있음 |

---

## 정리

EIP-1155는 `Multi Token Standard`다.
하나의 컨트랙트에서 여러 종류의 토큰을 관리하고, 대체 가능 토큰과 대체 불가능 토큰을 함께 표현할 수 있다.

EIP-1155가 해결한 문제는 다음과 같다.

```text
토큰 종류마다 컨트랙트가 늘어나는 문제
여러 토큰을 한 번에 전송하기 어려운 문제
게임처럼 다양한 자산을 함께 다뤄야 하는 문제
ERC-20과 ERC-721을 따로 조합해야 하는 비효율
```

다만 모든 상황에서 ERC-1155가 정답은 아니다.
단순 코인이라면 ERC-20이 적합하고, 완전히 고유한 NFT 컬렉션이라면 ERC-721이 더 직관적일 수 있다.

ERC-1155는 여러 종류의 자산을 효율적으로 묶어 다뤄야 할 때 강력하다.
