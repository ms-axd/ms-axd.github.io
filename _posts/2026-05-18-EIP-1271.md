EIP-1271은 스마트 컨트랙트 지갑이 서명을 검증하는 방법을 정의한 표준이다.

정식 문서 기준으로는 `ERC-1271`이다.

핵심은 단순하다.

```text
EOA는 ecrecover로 서명자를 확인한다.
스마트 컨트랙트 지갑은 isValidSignature로 서명이 유효한지 확인한다.
```

즉, EIP-1271은 이런 질문에 답한다.

```text
이 스마트 컨트랙트 지갑이
이 서명을 유효하다고 인정하는가?
```

---

## EOA 서명 검증

Ethereum에서 일반 지갑은 `EOA`다.

EOA는 Externally Owned Account의 약자다.

쉽게 말하면 개인키로 직접 제어되는 계정이다.

여기서 자주 나오는 함수가 `ecrecover`다.

`ecrecover`는 메시지 해시와 서명값을 받아서,
그 서명을 만든 주소를 복구하는 EVM 내장 함수다.

정확히는 ECDSA 서명값인 `v`, `r`, `s`와 메시지 해시를 사용한다.

EOA가 메시지에 서명하면,
검증자는 `ecrecover`를 사용해서 서명자를 복구할 수 있다.

```text
message
-> hash
-> signature
-> ecrecover
-> signer address
```

복구된 주소가 예상한 지갑 주소와 같으면 서명이 유효하다고 판단한다.

```text
복구된 주소 == 기대한 주소
-> 유효한 서명
```

이 방식은 EOA에는 잘 맞는다.

EOA는 하나의 개인키가 계정을 대표하기 때문이다.

---

## 스마트 컨트랙트 지갑의 문제

스마트 컨트랙트 지갑은 다르다.

스마트 컨트랙트 지갑은 개인키 하나로만 움직이지 않는다.

예를 들어 멀티시그 지갑은 여러 명의 승인이 필요할 수 있다.

```text
Alice 서명
Bob 서명
Charlie 서명

2명 이상 승인하면 실행
```

또 어떤 지갑은 권한 정책을 가질 수 있다.

```text
관리자 키
일일 한도 키
특정 dApp 전용 키
일정 시간 동안만 유효한 세션 키
```

이런 지갑은 단순히 `ecrecover`로 주소 하나를 복구한다고 끝나지 않는다.

중요한 것은 서명자 주소 하나가 아니다.

그 지갑의 내부 규칙이 이 서명을 유효하다고 인정하는지다.

---

## 왜 ecrecover만으로 부족한가

`ecrecover`는 ECDSA 서명에서 주소를 복구한다.

하지만 스마트 컨트랙트 지갑에는 고정된 개인키가 없다.

컨트랙트는 코드로 동작한다.

그래서 서명 검증도 코드의 판단을 따라야 한다.

```text
EOA
-> 개인키가 계정을 대표
-> ecrecover로 검증

스마트 컨트랙트 지갑
-> 컨트랙트 로직이 계정을 대표
-> 컨트랙트에 직접 물어봐야 함
```

이때 필요한 표준 인터페이스가 EIP-1271이다.

---

## isValidSignature

EIP-1271의 핵심 함수는 `isValidSignature`다.

```solidity
function isValidSignature(
    bytes32 hash,
    bytes memory signature
) public view returns (bytes4 magicValue);
```

검증자는 스마트 컨트랙트 지갑에 직접 묻는다.

```text
이 hash에 대한 이 signature가 유효한가?
```

스마트 컨트랙트 지갑은 자기 내부 규칙에 따라 판단한다.

```text
멀티시그 조건을 만족했는가?
허용된 서명자인가?
서명이 만료되지 않았는가?
정책상 이 요청을 허용하는가?
```

유효하면 정해진 값을 반환한다.

유효하지 않으면 다른 값을 반환하거나 revert할 수 있다.

---

## Magic Value

EIP-1271에서 유효한 서명은 특정 값을 반환해야 한다.

```text
0x1626ba7e
```

이 값을 `magic value`라고 부른다.

검증자는 반환값이 이 값인지 확인한다.

```text
isValidSignature(hash, signature)
-> 0x1626ba7e
-> 유효한 서명
```

왜 boolean이 아니라 magic value를 쓸까?

실수로 true처럼 보이는 값을 반환하는 상황을 줄이기 위해서다.

정확히 약속된 값이 돌아와야만 유효한 서명으로 인정한다.

---

## 검증 흐름

dApp이나 컨트랙트가 서명을 검증할 때는 계정 종류에 따라 방식이 달라진다.

```text
서명자 주소가 EOA인가?
-> ecrecover로 검증

서명자 주소가 컨트랙트인가?
-> isValidSignature 호출
```

흐름을 조금 더 풀면 이렇다.

```text
1. 사용자가 메시지에 서명한다.
2. dApp이 서명자 주소를 확인한다.
3. 주소에 코드가 없으면 EOA로 보고 ecrecover를 사용한다.
4. 주소에 코드가 있으면 스마트 컨트랙트 지갑으로 보고 isValidSignature를 호출한다.
5. 반환값이 0x1626ba7e이면 유효한 서명으로 처리한다.
```

EIP-1271은 이 중 4번과 5번을 표준화한다.

---

## 멀티시그 지갑 예시

멀티시그 지갑을 생각해보자.

이 지갑은 세 명의 소유자가 있고,
그중 두 명 이상이 서명해야 유효하다고 하자.

```text
owners = [Alice, Bob, Charlie]
threshold = 2
```

Alice만 서명하면 부족하다.

```text
Alice 서명
-> 1명 승인
-> 유효하지 않음
```

Alice와 Bob이 서명하면 조건을 만족한다.

```text
Alice 서명 + Bob 서명
-> 2명 승인
-> 유효함
-> 0x1626ba7e 반환
```

이 판단은 외부에서 단순히 `ecrecover` 한 번으로 처리할 수 없다.

멀티시그 조건은 지갑 컨트랙트 내부에 있기 때문이다.

그래서 외부 컨트랙트는 지갑에게 직접 물어봐야 한다.

```text
이 서명 묶음이 너의 규칙상 유효한가?
```

그 질문의 표준 형태가 `isValidSignature`다.

---

## 왜 중요한가

EIP-1271은 스마트 컨트랙트 지갑을 일반 지갑처럼 사용할 수 있게 만든다.

서명이 필요한 곳은 많다.

```text
로그인
오프체인 주문
NFT 거래 승인
토큰 권한 위임
거버넌스 투표
메시지 기반 인증
```

EOA만 지원하면 스마트 컨트랙트 지갑 사용자는 이런 기능을 제대로 쓰기 어렵다.

반대로 EIP-1271을 지원하면 dApp은 EOA와 스마트 컨트랙트 지갑을 함께 처리할 수 있다.

```text
EOA 사용자
-> ecrecover

스마트 컨트랙트 지갑 사용자
-> isValidSignature
```

지갑의 구조가 달라도 dApp 입장에서는 표준 방식으로 서명을 확인할 수 있다.

---

## EIP-4337과의 연결

EIP-4337은 계정 추상화를 구현하는 대표적인 방식이다.

여기서 사용자는 스마트 컨트랙트 계정을 지갑처럼 사용한다.

그런데 스마트 컨트랙트 계정이 지갑처럼 쓰이려면 서명 검증 방식도 표준화되어야 한다.

```text
EIP-4337
-> 스마트 컨트랙트 계정을 지갑처럼 사용

EIP-1271
-> 스마트 컨트랙트 계정의 서명 검증 방식 제공
```

즉, EIP-1271은 계정 추상화의 기반 지식에 가깝다.

스마트 계정이 어떤 방식으로 서명을 인정하는지 이해해야 EIP-4337도 이해하기 쉽다.

---

## EIP-7702와의 연결

EIP-7702는 기존 EOA가 일시적으로 스마트 컨트랙트 코드처럼 동작할 수 있게 한다.

이 흐름에서도 중요한 질문은 비슷하다.

```text
계정의 동작 방식이 단순 EOA에서 스마트 계정 쪽으로 이동하면
서명 검증은 누가, 어떤 규칙으로 판단하는가?
```

EIP-1271은 이 질문의 기본 형태를 먼저 보여준다.

서명 검증을 개인키 하나에만 고정하지 않고,
계정의 로직이 직접 판단하게 만드는 방식이다.

그래서 EIP-1271을 먼저 보면 EIP-4337과 EIP-7702를 이해하기 쉽다.

---

## 주의할 점

EIP-1271은 서명 검증의 인터페이스를 정한다.

하지만 어떤 서명을 유효하다고 볼지는 각 지갑 구현이 책임진다.

```text
표준이 보장하는 것
-> 검증 함수의 형태
-> 유효할 때 반환해야 하는 값

표준이 보장하지 않는 것
-> 지갑 내부 정책의 안전성
-> 서명자 관리의 안전성
-> 멀티시그 구현의 정확성
```

따라서 `isValidSignature`가 있다고 해서 자동으로 안전한 지갑이 되는 것은 아니다.

검증 로직이 잘못되면 잘못된 서명도 유효하다고 판단할 수 있다.

---

## 정리

EIP-1271은 스마트 컨트랙트 지갑의 서명 검증 표준이다.

EOA는 `ecrecover`로 서명자를 확인할 수 있다.

하지만 스마트 컨트랙트 지갑은 내부 로직에 따라 서명의 유효성을 판단한다.

그래서 외부에서는 지갑 컨트랙트의 `isValidSignature`를 호출해야 한다.

```text
EOA
-> ecrecover

스마트 컨트랙트 지갑
-> isValidSignature
```

유효한 서명이면 지갑은 `0x1626ba7e`를 반환한다.

이 표준 덕분에 dApp은 EOA뿐 아니라 멀티시그 지갑, 스마트 계정, 계정 추상화 지갑도 같은 흐름에서 지원할 수 있다.
