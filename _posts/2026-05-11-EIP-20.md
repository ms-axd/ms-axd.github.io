우선 EIP가 뭔지 알아보자.

EIP는 단순한 기능 명세가 아니다.
이더리움 생태계에서 반복해서 생기는 문제를 표준화된 방식으로 풀기 위한 합의 문서다.
그래서 EIP를 읽을 때는 함수 이름이나 파라미터만 보는 것보다, 그 제안이 나온 배경과 사용 맥락을 같이 봐야 한다.

## EIP란?

`Ethereum Improvement Proposal`

이더리움의 기능 추가, 규칙 변경, 표준 정의 등을 제안하는 공식 문서 체계다.

쉽게 말하면

> "이더리움을 이렇게 바꾸자"

를 정리해서 올리는 설계 문서이다.

### 왜 중요한가

이더리움은 중앙 관리자 없이 운영된다.

그래서 누가 기능 하나 추가하려 해도

```text
- 왜 필요한지
- 어떻게 동작하는지
- 기존 시스템과 충돌 없는지
- 보안 문제 없는지
```

이런 것들을 공개적으로 검토해야 한다. 

이걸 위한 공식 제안 시스템이 `EIP`이다.

### EIP 구조

보통 이런 형식으로 작성된다.

```text
EIP 번호
제목
작성자
상태
종류
내용
동기
기술 명세
보안 고려사항
```

- EIP-20 -> ERC-20 토큰 표준
- EIP-721 -> NFT 표준
- EIP-1559 -> 수수료 구조 변경


#### EIP 와 ERC의 차이

| 구분 | 의미 |
| --- | --- |
| EIP | 이더리움 개선 제안 전체 |
| ERC | 주로 토큰/스마트컨트랙트 표준 |

ERC도 EIP의 한 종류다

정확히는 ERC는 `Standards Track EIP` 안에 포함되는 하위 분류에 가깝다.
이더리움 네트워크 자체를 바꾸는 제안도 EIP이고, 스마트 컨트랙트가 따라야 할 인터페이스 표준도 EIP이다.
다만 후자는 보통 ERC라고 부른다.

---

## EIP의 종류

EIP는 목적에 따라 크게 나뉜다.

| 종류 | 의미 | 예시 |
| --- | --- | --- |
| Standards Track | 이더리움 구현체, 프로토콜, 표준에 영향을 주는 제안 | EIP-20, EIP-1559 |
| Meta | EIP 프로세스나 개발 절차에 대한 제안 | EIP-1 |
| Informational | 정보 공유 목적의 문서 | 설계 설명, 권장사항 |

`Standards Track`은 다시 세부 분류가 있다.

| 분류 | 설명 |
| --- | --- |
| Core | 합의 규칙, 블록 처리, 네트워크 프로토콜처럼 이더리움 코어에 영향을 주는 변경 |
| Networking | devp2p, 네트워크 메시지 같은 통신 계층 변경 |
| Interface | RPC API, ABI, 지갑 인터페이스 같은 외부 인터페이스 |
| ERC | 애플리케이션 레벨 표준. 토큰, NFT, 스마트 컨트랙트 인터페이스 등 |

여기서 우리가 자주 듣는 ERC-20, ERC-721, ERC-1155는 전부 ERC 계열 EIP이다.

---

## EIP 상태

EIP는 그냥 작성하고 끝나는 문서가 아니다.
상태가 있고, 검토를 거치면서 바뀐다.

| 상태 | 의미 |
| --- | --- |
| Draft | 초안 상태 |
| Review | 커뮤니티 검토 단계 |
| Last Call | 최종 검토 단계 |
| Final | 확정된 상태 |
| Stagnant | 오래 갱신되지 않은 상태 |
| Withdrawn | 작성자가 철회한 상태 |
| Living | 계속 갱신되는 문서 |

중요한 점은 `Final`이라고 해서 모든 클라이언트가 바로 적용했다는 뜻은 아니다.
EIP는 표준 문서이고, 실제 적용은 클라이언트 구현, 하드포크, 생태계 채택 같은 과정이 따로 필요하다.

---

## EIP가 만들어지는 흐름

일반적인 흐름은 다음과 같다.

```text
1. 문제를 정의한다.
2. 해결 방식을 문서로 작성한다.
3. EIP 저장소에 PR을 올린다.
4. 에디터와 커뮤니티가 형식과 내용을 검토한다.
5. 논의 결과에 따라 수정한다.
6. 충분히 합의되면 상태가 올라간다.
7. 필요한 경우 클라이언트나 스마트 컨트랙트 생태계에서 구현한다.
```

핵심은 코드보다 먼저 합의 가능한 명세를 만든다는 점이다.
블록체인은 한 번 배포되면 되돌리기 어렵기 때문에, 변경 전에 공개 검토가 중요하다.

## EIP-20이란?

`EIP-20`은 흔히 `ERC-20`이라고 부르는 토큰 표준이다.

이 표준은 이더리움 위에서 동작하는 대체 가능 토큰의 공통 인터페이스를 정의한다.
대체 가능하다는 말은 각 토큰 단위가 서로 같은 가치를 가진다는 뜻이다.

예를 들어 A가 가진 1 USDT와 B가 가진 1 USDT는 구분할 필요가 없다.
둘 다 같은 1 USDT이다.
이런 토큰을 만들 때 가장 널리 쓰이는 표준이 ERC-20이다.

---

## EIP-20은 왜 등장했나

이더리움은 스마트 컨트랙트를 통해 누구나 토큰을 만들 수 있게 했다.
하지만 자유도가 높다는 것은 표준이 없으면 각자 다른 방식으로 만든다는 뜻이기도 하다.

토큰 A는 잔액 조회 함수를 `balanceOf`로 만들고, 토큰 B는 `getBalance`로 만들고, 토큰 C는 전송 함수를 `sendToken`으로 만든다고 해보자.
그러면 지갑, 거래소, 블록 익스플로러, 다른 스마트 컨트랙트는 토큰마다 별도 코드를 작성해야 한다.

이 구조는 확장성이 없다.
토큰이 많아질수록 연동 비용이 폭발적으로 증가한다.

EIP-20은 이 문제에서 출발했다.


> 토큰을 만드는 방식은 자유롭게 두되,
> 외부에서 토큰을 다루는 최소 인터페이스는 통일하자.


즉 EIP-20의 핵심은 토큰 구현을 강제하는 것이 아니라, 토큰과 외부 시스템이 대화하는 공통 규칙을 정하는 것이다.

### ERC-20이 필요한 이유

ERC-20 이전에도 스마트 컨트랙트로 토큰을 만들 수는 있었다.
문제는 각 토큰마다 함수 이름과 동작 방식이 다르면 지갑, 거래소, DeFi 프로토콜이 매번 따로 연동해야 한다는 점이다.

ERC-20은 이 문제를 해결했다.

```text
토큰 잔액 조회
토큰 전송
제3자 전송 허용
허용량 조회
전송 이벤트 기록
```

이런 기본 기능을 같은 이름과 같은 규칙으로 제공하도록 정했다.
그 결과 지갑이나 거래소는 ERC-20 인터페이스만 지원하면 수많은 토큰을 공통 방식으로 다룰 수 있다.

---

## EIP-20이 해결한 문제

EIP-20이 해결한 문제는 크게 세 가지다.

### 1. 토큰 호환성 문제

표준이 없으면 토큰마다 사용법이 달라진다.
지갑은 각 토큰의 잔액을 어떻게 읽어야 하는지 알아야 하고, 거래소는 전송 함수를 토큰마다 따로 처리해야 한다.

ERC-20은 이 문제를 `balanceOf`, `transfer`, `approve`, `transferFrom` 같은 공통 함수로 해결했다.

### 2. 생태계 연동 비용 문제

토큰 하나를 새로 만들 때마다 지갑, 거래소, DeFi 서비스가 별도 연동을 해야 한다면 생태계는 빠르게 커질 수 없다.

ERC-20을 따르면 새 토큰도 기존 도구와 바로 호환될 수 있다.
이 점이 중요하다.
ERC-20은 토큰 개발자만 편하게 만든 표준이 아니라, 토큰을 둘러싼 전체 인프라의 연동 비용을 낮춘 표준이다.

### 3. 스마트 컨트랙트 간 조합성 문제

이더리움의 강점은 여러 스마트 컨트랙트가 서로 호출되며 새로운 서비스를 만들 수 있다는 점이다.
이를 보통 `composability`라고 부른다.

ERC-20이 없었다면 DeFi 프로토콜은 토큰마다 다른 전송 방식을 처리해야 한다.
하지만 ERC-20이 있으면 대출, 스왑, 예치, 스테이킹 컨트랙트가 같은 방식으로 토큰을 받을 수 있다.

이 표준화가 있었기 때문에 DeFi 생태계가 빠르게 확장될 수 있었다.

## ERC-20 핵심 함수

ERC-20에서 중요한 함수는 다음과 같다.

| 함수 | 설명 |
| --- | --- |
| `totalSupply()` | 전체 발행량 조회 |
| `balanceOf(address)` | 특정 주소의 잔액 조회 |
| `transfer(address,uint256)` | 내 토큰을 다른 주소로 전송 |
| `approve(address,uint256)` | 특정 주소가 내 토큰을 대신 쓸 수 있도록 허용 |
| `allowance(address,address)` | 허용된 사용량 조회 |
| `transferFrom(address,address,uint256)` | 허용량을 사용해 제3자가 토큰 전송 |

실제 인터페이스는 대략 이런 형태다.

```solidity
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
    function allowance(address owner, address spender) external view returns (uint256);
    function approve(address spender, uint256 amount) external returns (bool);
    function transferFrom(address from, address to, uint256 amount) external returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 value);
    event Approval(address indexed owner, address indexed spender, uint256 value);
}
```

### transfer

`transfer`는 사용자가 자기 토큰을 직접 보내는 함수다.

```solidity
transfer(receiver, 100);
```

이 호출은 호출자의 잔액에서 `receiver`의 잔액으로 100만큼 이동시킨다.

### approve

`approve`는 다른 주소에게 내 토큰을 대신 사용할 권한을 주는 함수다.
DeFi에서 매우 자주 쓰인다.

예를 들어 사용자가 DEX에 토큰 교환을 맡기려면 DEX 컨트랙트가 사용자의 토큰을 가져갈 수 있어야 한다.
이때 사용자는 먼저 DEX 컨트랙트에 allowance를 준다.

```solidity
approve(dexAddress, 1000);
```

이제 `dexAddress`는 최대 1000만큼 사용자의 토큰을 `transferFrom`으로 이동시킬 수 있다.

### transferFrom

`transferFrom`은 허용된 권한을 사용해 토큰을 전송한다.

```solidity
transferFrom(user, pool, 500);
```

이 함수는 보통 거래소, 스테이킹 컨트랙트, DeFi 프로토콜에서 사용된다.

---

## 실제로 어디에서 사용되는가

ERC-20은 이더리움 생태계의 기본 단위처럼 쓰인다.

### 지갑

MetaMask 같은 지갑은 ERC-20 인터페이스를 통해 토큰 잔액을 조회하고 전송한다.
지갑이 모든 토큰의 내부 구현을 알 필요는 없다.
`balanceOf`와 `Transfer` 이벤트를 기준으로 표시하면 된다.

### 거래소

중앙화 거래소와 탈중앙화 거래소 모두 ERC-20 표준을 기반으로 토큰 입출금과 거래를 처리한다.
특히 DEX에서는 사용자가 먼저 `approve`로 토큰 사용 권한을 주고, 이후 컨트랙트가 `transferFrom`으로 토큰을 가져가는 방식이 일반적이다.

### DeFi

스왑, 렌딩, 유동성 풀, 스테이킹 같은 DeFi 서비스는 대부분 ERC-20 토큰을 기본 자산으로 다룬다.

예를 들어 사용자가 토큰을 예치하면 컨트랙트는 다음 흐름을 사용한다.

```text
1. 사용자가 DeFi 컨트랙트에 approve 실행
2. DeFi 컨트랙트가 transferFrom으로 토큰 수령
3. 내부 장부에 사용자의 예치량 기록
4. 보상, 대출, 스왑 같은 후속 로직 실행
```

여기서 ERC-20 표준이 없으면 1번과 2번이 토큰마다 달라진다.
그러면 DeFi 컨트랙트는 범용적으로 동작하기 어렵다.

### 블록 익스플로러와 인덱서

Etherscan 같은 블록 익스플로러는 `Transfer` 이벤트를 추적해서 토큰 이동 내역을 보여준다.
The Graph 같은 인덱싱 시스템도 이벤트 로그를 기반으로 데이터를 쌓는다.

즉 ERC-20은 단순히 컨트랙트 개발자만을 위한 표준이 아니다.
지갑, 거래소, DeFi, 분석 도구, 인덱서가 모두 의존하는 공통 규칙이다.

---

## ERC-20 이벤트

ERC-20에는 중요한 이벤트가 두 개 있다.

| 이벤트 | 의미 |
| --- | --- |
| `Transfer` | 토큰 전송 발생 |
| `Approval` | 토큰 사용 권한 승인 발생 |

이벤트는 블록체인 로그에 기록된다.
지갑, 블록 익스플로러, 인덱서가 토큰 이동 내역을 추적할 때 이 이벤트를 사용한다.

특히 `Transfer` 이벤트는 단순한 로그가 아니라 생태계 호환성에 중요하다.
이벤트를 제대로 발생시키지 않으면 지갑이나 익스플로러에서 잔액과 전송 내역이 이상하게 보일 수 있다.

---

## ERC-20의 한계

ERC-20은 단순하고 널리 쓰이지만 완벽하지 않다.

### approve race condition

`approve`에는 오래된 문제점이 있다.

사용자가 기존 allowance를 100에서 50으로 줄이려 할 때, spender가 먼저 100을 사용하고 이후 50이 다시 승인되는 상황이 생길 수 있다.
그러면 의도보다 많은 토큰이 사용될 수 있다.

그래서 실무에서는 다음 방식이 자주 쓰인다.

```text
1. allowance를 먼저 0으로 설정
2. 새 allowance를 다시 설정
```

또는 `increaseAllowance`, `decreaseAllowance` 같은 확장 함수를 사용한다.

### 토큰 수신 감지 문제

ERC-20의 `transfer`는 받는 주소가 일반 지갑인지 컨트랙트인지 구분하지 않는다.
만약 토큰을 받을 수 없는 컨트랙트 주소로 전송하면 토큰이 사실상 잠길 수 있다.

이 문제를 개선하려고 ERC-777, ERC-1363 같은 다른 표준도 등장했다.

### 구현체별 차이

일부 오래된 토큰은 ERC-20 표준을 완전히 지키지 않는다.
예를 들어 `transfer`가 `bool`을 반환하지 않는 토큰도 있다.
그래서 실무에서는 OpenZeppelin의 `SafeERC20` 같은 래퍼를 사용해 호환성 문제를 줄인다.

---

## EIP-20의 의미

EIP-20의 가치는 단순히 토큰 함수를 정했다는 데 있지 않다.
더 중요한 점은 이더리움 토큰 생태계의 공통 언어를 만들었다는 것이다.

ERC-20 덕분에 다음이 가능해졌다.

- 지갑이 여러 토큰을 공통 방식으로 표시
- 거래소가 새 토큰을 쉽게 상장
- DeFi 프로토콜이 토큰을 표준 방식으로 처리
- 블록 익스플로러가 전송 내역을 일관되게 추적
- 개발자가 검증된 인터페이스를 기반으로 토큰 발행

즉 EIP-20은 이더리움 애플리케이션 생태계가 폭발적으로 커지는 데 핵심 역할을 했다.

분석 관점에서 보면 EIP-20은 다음 의미를 가진다.

| 관점 | 의미 |
| --- | --- |
| 기술적 관점 | 토큰 인터페이스를 통일했다 |
| 생태계 관점 | 지갑, 거래소, DeFi의 연동 비용을 낮췄다 |
| 보안 관점 | 표준화된 패턴을 만들었지만 approve 계열 위험도 남겼다 |
| 경제적 관점 | 누구나 호환 가능한 토큰을 발행할 수 있게 했다 |

---

## 정리

EIP-20은 단순한 토큰 함수 목록이 아니다.
이더리움 생태계가 토큰을 공통 방식으로 다루게 만든 표준이다.

ERC-20은 `balanceOf`, `transfer`, `approve`, `transferFrom` 같은 공통 인터페이스를 제공한다.
이 표준 덕분에 지갑, 거래소, DeFi, 블록 익스플로러가 같은 방식으로 토큰을 다룰 수 있게 됐다.

다만 `approve` 문제, 토큰 수신 감지 문제, 구현체별 차이 같은 한계도 있다.
그래서 실제 개발에서는 표준을 이해하는 것뿐 아니라 OpenZeppelin 같은 검증된 구현체를 사용하는 것이 중요하다.
