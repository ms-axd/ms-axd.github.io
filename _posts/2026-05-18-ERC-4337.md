ERC-4337은 Ethereum 프로토콜을 직접 바꾸지 않고 Account Abstraction을 구현하는 표준이다.

핵심은 단순하다.

```text
사용자는 일반 트랜잭션 대신 UserOperation을 만든다.
Bundler가 UserOperation을 모아 EntryPoint에 제출한다.
Smart Account가 자기 규칙으로 검증하고 실행한다.
```

즉, ERC-4337은 스마트 컨트랙트 지갑을 실제 계정처럼 쓰기 위한 구조다.

---

## 왜 필요한가

기존 Ethereum 계정은 크게 두 가지다.

```text
EOA
-> 개인키로 직접 제어되는 계정

Contract Account
-> 코드로 동작하는 계정
```

일반 사용자는 대부분 EOA를 쓴다.

EOA는 단순하지만 한계가 있다.

```text
개인키 하나가 계정 전체 권한을 가짐
가스비는 ETH로 직접 내야 함
트랜잭션을 여러 개 묶기 어려움
계정 복구 로직을 넣기 어려움
권한을 세분화하기 어려움
```

사용자 경험을 개선하려면 계정이 더 똑똑해야 한다.

예를 들어 이런 기능이 필요하다.

```text
소셜 복구
멀티시그
가스 대납
ERC-20 토큰으로 가스비 지불
여러 작업을 하나로 묶기
세션 키
일일 한도
```

이런 기능은 EOA보다 스마트 컨트랙트 계정에 더 잘 맞는다.

---

## Account Abstraction

Account Abstraction은 계정의 검증 로직을 고정된 프로토콜 규칙에서 분리하는 개념이다.

기존 EOA는 검증 방식이 정해져 있다.

```text
트랜잭션 서명
-> ECDSA 검증
-> nonce 확인
-> ETH로 가스비 지불
```

Account Abstraction에서는 계정이 직접 검증 규칙을 가진다.

```text
이 서명이 유효한가?
이 작업을 실행해도 되는가?
가스비는 누가 낼 것인가?
이 nonce는 허용되는가?
```

ERC-4337은 이 구조를 Ethereum 합의 계층 변경 없이 구현한다.

새로운 트랜잭션 타입을 만들지 않는다.

대신 `UserOperation`이라는 별도 객체를 사용한다.

---

## UserOperation

`UserOperation`은 사용자가 실행하고 싶은 작업을 담은 객체다.

이름은 트랜잭션과 비슷하지만 트랜잭션은 아니다.

```text
transaction
-> Ethereum 프로토콜이 직접 처리하는 트랜잭션

UserOperation
-> ERC-4337 인프라가 처리하는 pseudo-transaction
```

UserOperation에는 이런 정보가 들어간다.

```text
sender
nonce
callData
callGasLimit
verificationGasLimit
maxFeePerGas
maxPriorityFeePerGas
paymaster
signature
```

중요한 점은 `signature`의 의미를 프로토콜이 고정하지 않는다는 것이다.

서명을 어떻게 검증할지는 Smart Account가 정한다.

```text
EOA
-> 프로토콜이 서명 검증 규칙을 정함

Smart Account
-> 계정 컨트랙트가 서명 검증 규칙을 정함
```

이 부분이 Account Abstraction의 핵심이다.

---

## Bundler

사용자는 UserOperation을 직접 블록에 넣을 수 없다.

UserOperation은 Ethereum 트랜잭션이 아니기 때문이다.

그래서 `Bundler`가 필요하다.

Bundler는 UserOperation을 모아서 하나의 실제 트랜잭션으로 만든다.

```text
UserOperation A
UserOperation B
UserOperation C

-> Bundler
-> EntryPoint.handleOps()
-> Ethereum transaction
```

Bundler는 일반 트랜잭션을 보내는 주체다.

사용자 대신 UserOperation을 체인에 올리는 역할을 한다.

---

## EntryPoint

`EntryPoint`는 ERC-4337의 중심 컨트랙트다.

Bundler는 UserOperation 묶음을 EntryPoint에 제출한다.

핵심 함수는 `handleOps`다.

```solidity
function handleOps(
    PackedUserOperation[] calldata ops,
    address payable beneficiary
) external;
```

EntryPoint는 각 UserOperation에 대해 두 단계를 처리한다.

```text
1. 검증
2. 실행
```

검증 단계에서는 Smart Account에 묻는다.

```text
이 UserOperation이 유효한가?
서명은 맞는가?
nonce는 맞는가?
가스비를 낼 수 있는가?
```

실행 단계에서는 사용자가 요청한 작업을 수행한다.

```text
토큰 전송
NFT 구매
컨트랙트 호출
여러 작업 묶음 실행
```

---

## Smart Account

ERC-4337에서 사용자의 계정은 Smart Account다.

Smart Account는 최소한 UserOperation을 검증할 수 있어야 한다.

대표 인터페이스는 `validateUserOp`다.

```solidity
function validateUserOp(
    PackedUserOperation calldata userOp,
    bytes32 userOpHash,
    uint256 missingAccountFunds
) external returns (uint256 validationData);
```

이 함수는 계정의 규칙에 따라 UserOperation을 검증한다.

```text
서명이 유효한가?
허용된 키인가?
세션 키가 만료되지 않았는가?
지출 한도를 넘지 않았는가?
필요한 가스비를 낼 수 있는가?
```

EOA에서는 이런 검증 로직을 넣을 수 없다.

Smart Account에서는 가능하다.

---

## Paymaster

`Paymaster`는 가스비를 대신 내주는 컨트랙트다.

ERC-4337에서 Paymaster를 사용하면 사용자가 ETH를 직접 들고 있지 않아도 된다.

```text
사용자
-> UserOperation 생성

Paymaster
-> 가스비 지불 동의

Bundler
-> 트랜잭션 제출
```

이 구조로 이런 UX가 가능해진다.

```text
dApp이 신규 사용자의 가스비를 대신 냄
사용자가 ERC-20 토큰으로 수수료를 지불
특정 조건을 만족할 때만 가스 대납
```

Paymaster는 편리하지만 위험도 있다.

검증 로직이 약하면 악의적인 UserOperation에 비용을 낭비할 수 있다.

---

## Factory

사용자의 Smart Account가 아직 배포되지 않았을 수도 있다.

ERC-4337은 계정 생성도 UserOperation 흐름에 포함할 수 있다.

이때 `Factory`가 사용된다.

```text
아직 계정 없음
-> UserOperation에 생성 정보 포함
-> Factory가 Smart Account 배포
-> 같은 흐름에서 작업 실행
```

사용자 입장에서는 계정을 미리 배포하는 과정을 덜 의식하게 된다.

---

## 전체 흐름

ERC-4337의 전체 흐름은 이렇게 볼 수 있다.

```text
1. 사용자가 UserOperation을 만든다.
2. UserOperation을 별도 mempool로 보낸다.
3. Bundler가 여러 UserOperation을 모은다.
4. Bundler가 EntryPoint.handleOps() 트랜잭션을 보낸다.
5. EntryPoint가 Smart Account의 validateUserOp를 호출한다.
6. 검증에 성공하면 실제 callData를 실행한다.
7. 가스비는 Smart Account 또는 Paymaster가 처리한다.
```

이 구조 덕분에 Ethereum 프로토콜을 바꾸지 않고도 스마트 계정 경험을 만들 수 있다.

---

## EIP-1271과의 연결

EIP-1271은 스마트 컨트랙트 지갑이 서명을 검증하는 표준이다.

ERC-4337은 Smart Account가 UserOperation을 검증하는 구조다.

둘은 관심사가 다르지만 연결된다.

```text
EIP-1271
-> 이 스마트 컨트랙트 지갑이 이 서명을 유효하다고 보는가?

ERC-4337
-> 이 Smart Account가 이 UserOperation을 실행해도 되는가?
```

ERC-4337 계정은 내부적으로 EIP-1271을 지원할 수 있다.

그러면 dApp은 Smart Account 사용자의 메시지 서명도 표준 방식으로 검증할 수 있다.

---

## EIP-7702와의 연결

EIP-7702는 EOA가 코드 위임을 통해 스마트 계정처럼 동작할 수 있게 한다.

ERC-4337은 처음부터 Smart Account를 중심으로 설계된다.

```text
ERC-4337
-> 별도 UserOperation 인프라로 Smart Account 사용

EIP-7702
-> 기존 EOA에 코드 위임을 추가
```

둘 다 지갑 UX 개선을 목표로 한다.

차이는 접근 방식이다.

ERC-4337은 프로토콜 변경 없이 인프라 계층에서 풀고,
EIP-7702는 새로운 트랜잭션 타입으로 EOA 자체의 동작을 확장한다.

---

## 주의할 점

ERC-4337은 강력하지만 구조가 복잡하다.

등장하는 역할이 많다.

```text
UserOperation
Bundler
EntryPoint
Smart Account
Paymaster
Factory
Aggregator
```

그리고 보안 책임이 여러 곳으로 나뉜다.

```text
Smart Account 검증 로직
Paymaster 정책
Bundler 시뮬레이션
EntryPoint 호환성
nonce 관리
서명 검증 방식
```

Account Abstraction은 단순히 UX 기능을 추가하는 문제가 아니다.

계정의 보안 모델 자체를 코드로 옮기는 일이다.

---

## 정리

ERC-4337은 Account Abstraction을 Ethereum 프로토콜 변경 없이 구현하는 표준이다.

사용자는 트랜잭션 대신 `UserOperation`을 만든다.

Bundler는 UserOperation을 모아 EntryPoint에 제출한다.

EntryPoint는 Smart Account의 `validateUserOp`를 통해 검증하고 실행한다.

```text
UserOperation
-> Bundler
-> EntryPoint
-> Smart Account 검증
-> 실행
```

이 구조를 통해 스마트 컨트랙트 지갑은 EOA보다 유연한 계정 기능을 제공할 수 있다.
