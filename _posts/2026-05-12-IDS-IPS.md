---

둘 다 네트워크와 시스템을 보호하기 위한 보안 장비 또는 보안 솔루션이다.

핵심 차이는 단순하다.

```text
IDS -> 침입을 탐지하고 알림을 보낸다.
IPS -> 침입을 탐지하고 차단까지 수행한다.
```

IDS는 감시자에 가깝고, IPS는 감시자이면서 차단자에 가깝다.

---

## IDS란?

`IDS`는 `Intrusion Detection System`의 약자다.

침입 탐지 시스템이라고 부른다.

IDS는 네트워크 트래픽, 시스템 로그, 파일 변경, 프로세스 실행 같은 정보를 분석해서 공격 징후를 찾는다.

공격으로 판단되는 행위를 발견하면 관리자에게 알림을 보낸다.

```text
트래픽 또는 로그 수집
-> 공격 패턴 분석
-> 의심 행위 탐지
-> 관리자에게 경고
```

IDS의 목적은 공격을 직접 막는 것이 아니다.

공격이 발생했거나 발생 중이라는 사실을 빠르게 알려주는 것이 핵심이다.

예를 들어 다음과 같은 상황을 탐지할 수 있다.

| 탐지 대상 | 설명 |
| --- | --- |
| 포트 스캔 | 공격자가 열려 있는 서비스를 찾는 행위 |
| 악성 트래픽 | 알려진 공격 패턴과 유사한 패킷 |
| 비정상 로그인 | 짧은 시간 안에 반복되는 로그인 실패 |
| 권한 상승 시도 | 일반 사용자가 관리자 권한을 얻으려는 행위 |
| 파일 위변조 | 중요 파일이 예상과 다르게 변경되는 행위 |

---

## IPS란?

`IPS`는 `Intrusion Prevention System`의 약자다.

침입 방지 시스템이라고 부른다.

IPS는 IDS처럼 공격을 탐지한다.

하지만 탐지만 하지 않고 공격 트래픽을 직접 차단한다.

```text
트래픽 수집
-> 공격 여부 분석
-> 공격으로 판단
-> 패킷 차단 또는 연결 종료
```

IPS는 보통 네트워크 경로 중간에 배치된다.

트래픽이 실제 서버에 도달하기 전에 IPS를 통과하도록 만들고,
IPS가 위험하다고 판단한 요청을 차단한다.

예를 들어 다음과 같은 대응을 할 수 있다.

| 대응 방식 | 설명 |
| --- | --- |
| 패킷 차단 | 악성 패킷을 버린다 |
| 세션 종료 | 공격자의 연결을 강제로 끊는다 |
| IP 차단 | 특정 출발지 IP의 접근을 막는다 |
| 룰 기반 차단 | 사전에 정의한 정책에 따라 요청을 거부한다 |
| 속도 제한 | 과도한 요청을 제한해 DoS 공격을 완화한다 |

IPS는 공격을 실시간으로 막을 수 있다는 장점이 있다.

대신 잘못 판단하면 정상 트래픽까지 막을 수 있다.

---

## 동작 원리

IDS와 IPS는 기본적으로 비슷한 분석 방식을 사용한다.

차이는 분석 결과를 어떻게 처리하느냐에 있다.

```text
IDS: 분석 -> 경고
IPS: 분석 -> 경고 + 차단
```

주요 탐지 방식은 다음과 같다.

---

## 시그니처 기반 탐지

`시그니처 기반 탐지`는 이미 알려진 공격 패턴과 현재 트래픽을 비교하는 방식이다.

백신이 악성코드 패턴을 비교하는 방식과 비슷하다.

```text
공격 패턴 DB
-> 현재 트래픽과 비교
-> 일치하면 공격으로 판단
```

예를 들어 SQL Injection 공격 문자열, 특정 악성코드 통신 패턴, 취약점 공격 페이로드를 룰로 등록할 수 있다.

장점은 명확하다.

알려진 공격에 대해 탐지 정확도가 높고 분석 속도가 빠르다.

하지만 새로운 공격이나 변형 공격에는 약하다.

```text
장점: 알려진 공격 탐지에 강함
단점: 알려지지 않은 공격 탐지에 약함
```

---

## 이상 행위 기반 탐지

`이상 행위 기반 탐지`는 정상 상태를 기준으로 비정상적인 행위를 찾는 방식이다.

평소 트래픽 양, 로그인 패턴, 시스템 호출, 접속 시간 등을 학습하고,
그 기준에서 크게 벗어난 행위를 공격으로 의심한다.

```text
정상 패턴 학습
-> 현재 행위와 비교
-> 기준에서 벗어나면 의심
```

예를 들어 평소에는 하루 10회 로그인하던 계정이 갑자기 1분에 100회 로그인을 시도하면 이상 행위로 볼 수 있다.

이 방식은 알려지지 않은 공격을 탐지할 가능성이 있다.

하지만 정상과 비정상의 경계가 애매하면 오탐이 늘어난다.

```text
장점: 신규 공격 탐지 가능성
단점: 오탐 가능성 높음
```

---

## 정책 기반 탐지

`정책 기반 탐지`는 조직이 정한 보안 정책을 기준으로 위반 여부를 판단하는 방식이다.

예를 들어 다음과 같은 정책을 만들 수 있다.

```text
관리자 페이지는 사내 VPN에서만 접근 가능
DB 서버는 웹 서버에서만 접근 가능
업무 시간 외 SSH 접근은 경고
특정 국가 IP 대역은 차단
```

정책 기반 탐지는 조직 환경에 맞게 보안을 강제할 수 있다.

하지만 정책을 잘못 설계하면 업무에 필요한 트래픽까지 막을 수 있다.

---

## IDS의 유형

IDS는 어디를 감시하느냐에 따라 여러 유형으로 나뉜다.

대표적으로 `NIDS`와 `HIDS`가 있다.

---

### NIDS

`NIDS`는 `Network-based IDS`의 약자다.

네트워크 기반 침입 탐지 시스템이다.

스위치, 라우터, 방화벽 근처에서 네트워크 트래픽을 분석한다.

```text
사용자
-> 네트워크
-> NIDS가 트래픽 복사본 분석
-> 서버
```

NIDS는 여러 서버로 오가는 트래픽을 한 번에 볼 수 있다.

그래서 네트워크 전체의 공격 흐름을 파악하기 좋다.

하지만 암호화된 트래픽은 내용을 보기 어렵고,
서버 내부에서 발생한 행위는 놓칠 수 있다.

---

### HIDS

`HIDS`는 `Host-based IDS`의 약자다.

호스트 기반 침입 탐지 시스템이다.

서버나 PC에 설치되어 해당 장비 내부의 행위를 분석한다.

예를 들어 다음을 감시한다.

```text
로그인 기록
시스템 로그
파일 무결성
프로세스 실행
권한 변경
설정 파일 변경
```

HIDS는 서버 내부에서 발생하는 행위를 자세히 볼 수 있다.

하지만 각 서버마다 설치하고 관리해야 하므로 운영 부담이 커질 수 있다.

---

## IPS의 유형

IPS도 배치 위치와 보호 대상에 따라 나뉜다.

대표적으로 `NIPS`와 `HIPS`가 있다.

---

### NIPS

`NIPS`는 `Network-based IPS`의 약자다.

네트워크 경로 중간에 배치되어 악성 트래픽을 실시간으로 차단한다.

```text
사용자
-> NIPS
-> 정상 트래픽만 서버로 전달
```

NIPS는 외부에서 내부로 들어오는 공격을 막는 데 유용하다.

웹 서버, 메일 서버, API 서버 앞단에 배치할 수 있다.

하지만 모든 트래픽이 NIPS를 통과하므로 성능과 가용성이 중요하다.

NIPS 장애가 발생하면 네트워크 전체 장애로 이어질 수 있다.

---

### HIPS

`HIPS`는 `Host-based IPS`의 약자다.

개별 서버나 PC에 설치되어 내부 행위를 차단한다.

예를 들어 악성 프로세스 실행 차단, 중요 파일 변경 차단, 비정상 시스템 호출 차단을 수행할 수 있다.

HIPS는 엔드포인트 보안에 강하다.

하지만 서버마다 에이전트를 설치해야 하고,
업무 프로그램과 충돌하지 않도록 정책을 조정해야 한다.

---

## IDS와 IPS의 차이

IDS와 IPS의 가장 큰 차이는 대응 방식이다.

| 구분 | IDS | IPS |
| --- | --- | --- |
| 목적 | 침입 탐지 | 침입 탐지와 차단 |
| 동작 방식 | 트래픽 또는 로그 분석 후 경고 | 트래픽 분석 후 차단 |
| 배치 방식 | 보통 네트워크 옆에서 감시 | 보통 네트워크 경로 중간에 배치 |
| 정상 트래픽 영향 | 상대적으로 낮음 | 잘못 설정하면 정상 트래픽 차단 가능 |
| 대응 속도 | 관리자 대응 필요 | 자동 대응 가능 |
| 주요 위험 | 탐지만 하고 막지 못함 | 오탐 시 서비스 장애 가능 |

IDS는 공격을 알려주는 역할에 강하다.

IPS는 공격을 바로 막는 역할에 강하다.

그래서 둘 중 하나만 항상 정답이라고 보기 어렵다.

운영 환경에 따라 함께 사용하는 경우가 많다.

---

## 활용 방법

IDS와 IPS는 단독으로 쓰기보다 다른 보안 장비와 함께 사용할 때 효과가 크다.

대표적인 활용 방식은 다음과 같다.

---

### 방화벽과 함께 사용

방화벽은 IP, 포트, 프로토콜 기준으로 접근을 제어한다.

IDS와 IPS는 트래픽 내용과 행위를 분석한다.

```text
방화벽 -> 접근 가능 여부 판단
IDS/IPS -> 접근 내용이 공격인지 분석
```

예를 들어 방화벽이 `80`, `443` 포트를 허용하더라도,
그 안으로 들어오는 요청이 SQL Injection일 수 있다.

이때 IDS/IPS가 요청 내용을 분석해 탐지하거나 차단한다.

---

### SIEM과 연동

`SIEM`은 여러 보안 장비와 시스템 로그를 모아 분석하는 시스템이다.

IDS가 탐지한 이벤트를 SIEM으로 보내면,
방화벽 로그, 서버 로그, 인증 로그와 함께 분석할 수 있다.

```text
IDS 탐지 로그
+ 방화벽 로그
+ 서버 로그인 로그
+ 애플리케이션 로그
-> SIEM에서 통합 분석
```

이렇게 하면 단일 이벤트가 아니라 공격 흐름을 볼 수 있다.

---

### 서버 보호

웹 서버나 API 서버 앞에 IPS를 배치하면 공격 요청을 서버에 도달하기 전에 막을 수 있다.

대표적인 보호 대상은 다음과 같다.

```text
SQL Injection
XSS
디렉터리 트래버설
취약한 관리자 페이지 접근
무차별 대입 로그인 시도
악성 파일 업로드
```

다만 웹 공격 차단에는 WAF가 더 특화되어 있다.

따라서 네트워크 IPS와 WAF를 목적에 맞게 함께 사용하는 것이 좋다.

---

### 내부망 감시

IDS는 외부 공격뿐 아니라 내부망 이상 행위 감시에도 사용할 수 있다.

예를 들어 내부 PC가 악성코드에 감염되면 외부 C2 서버와 통신할 수 있다.

IDS는 이런 비정상 통신을 탐지해 감염 단말을 찾는 데 도움을 준다.

```text
내부 PC
-> 의심스러운 외부 서버와 반복 통신
-> IDS 탐지
-> 보안 담당자 조사
```

---

## 장점

IDS와 IPS의 장점은 다음과 같다.

| 구분 | 장점 |
| --- | --- |
| IDS | 서비스 영향이 적고 공격 분석에 유리하다 |
| IPS | 공격을 실시간으로 차단할 수 있다 |
| 공통 | 보안 이벤트를 기록하고 사고 대응 근거를 제공한다 |

IDS는 탐지와 분석에 강하다.

IPS는 실시간 방어에 강하다.

둘 다 보안 로그를 남기기 때문에 사고가 발생했을 때 원인 분석에도 도움이 된다.

---

## 단점

단점도 명확하다.

| 구분 | 단점 |
| --- | --- |
| IDS | 공격을 직접 막지 못한다 |
| IPS | 오탐 시 정상 서비스를 차단할 수 있다 |
| 공통 | 지속적인 룰 업데이트와 튜닝이 필요하다 |

IDS가 경고를 많이 보내도 담당자가 확인하지 않으면 의미가 줄어든다.

IPS는 자동 차단이 가능하지만,
잘못된 룰 하나가 서비스 장애로 이어질 수 있다.

따라서 IDS/IPS는 설치만으로 끝나는 장비가 아니다.

운영 환경에 맞게 계속 조정해야 한다.

---

## 오탐과 미탐

IDS/IPS 운영에서 중요한 개념이 `오탐`과 `미탐`이다.

```text
오탐: 정상 행위를 공격으로 판단
미탐: 공격 행위를 정상으로 판단
```

오탐이 많으면 보안 담당자가 불필요한 경고에 지친다.

IPS에서는 오탐이 더 위험하다.

정상 사용자의 요청을 차단해서 서비스 장애를 만들 수 있기 때문이다.

반대로 미탐이 많으면 실제 공격을 놓친다.

좋은 운영은 오탐과 미탐 사이의 균형을 잡는 것이다.

---

## 배치 예시

일반적인 웹 서비스 환경에서는 다음과 같이 배치할 수 있다.

```text
Internet
-> Firewall
-> IPS
-> WAF
-> Web Server
-> DB Server

IDS는 주요 구간의 트래픽 복사본을 받아 분석
HIDS는 중요 서버 내부 로그와 파일 변경 감시
```

방화벽은 기본 접근 제어를 맡는다.

IPS는 네트워크 공격을 차단한다.

WAF는 웹 요청의 공격 패턴을 더 세밀하게 본다.

IDS와 HIDS는 공격 흔적을 탐지하고 사고 분석에 필요한 로그를 제공한다.

---

## 운영할 때 주의할 점

IDS/IPS를 운영할 때는 다음을 신경 써야 한다.

```text
1. 최신 공격 패턴으로 룰을 업데이트한다.
2. 서비스 특성에 맞게 탐지 정책을 조정한다.
3. 오탐 로그를 분석해서 불필요한 경고를 줄인다.
4. IPS 차단 정책은 단계적으로 적용한다.
5. 장애 발생 시 우회 경로와 복구 절차를 준비한다.
6. 탐지 로그를 SIEM 또는 로그 시스템에 연동한다.
```

특히 IPS는 바로 차단하는 장비이므로 처음부터 강한 정책을 적용하면 위험하다.

처음에는 탐지 모드로 운영하면서 로그를 보고,
문제가 없는 룰부터 차단 모드로 전환하는 방식이 안정적이다.

## 정리

IDS와 IPS는 침입을 탐지하고 대응하기 위한 핵심 보안 시스템이다.

IDS는 공격을 발견하고 알려준다.

IPS는 공격을 발견하고 막는다.

```text
IDS = 탐지 중심
IPS = 차단 중심
```

IDS는 분석과 가시성에 강하고,
IPS는 실시간 방어에 강하다.

하지만 둘 다 완벽하지 않다.

알려지지 않은 공격, 암호화된 트래픽, 잘못된 정책, 과도한 오탐 같은 한계가 있다.

그래서 IDS/IPS는 방화벽, WAF, EDR, SIEM 같은 다른 보안 체계와 함께 운영해야 한다.

결국 중요한 것은 장비 자체가 아니라 운영이다.

공격 패턴을 계속 업데이트하고,
환경에 맞게 정책을 튜닝하고,
탐지된 이벤트를 실제 대응으로 연결해야 IDS/IPS가 제대로 동작한다.
