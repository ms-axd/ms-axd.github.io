## 1. 퍼징이란 무엇인가?

### 정의

퍼즈 테스팅(Fuzz testing) 또는 퍼징(Fuzzing)은 소프트웨어에 유효한 입력, 예상 밖의 입력, 무작위 입력을 자동으로 주입하고 실행 결과를 관찰하는 테스트 방식이다.

쉽게 말하면 프로그램에 이상한 입력값을 계속 던져서 버그를 찾는 기술이다. 보안 연구에서는 사람이 직접 찾기 어려운 입력 처리 버그를 대량으로 탐색하는 데 많이 사용한다.

퍼징에서 주로 감시하는 결과는 다음과 같다.

- 프로그램 크래시
- assertion failure
- 메모리 누수
- use-after-free, buffer overflow 같은 메모리 오류
- timeout 또는 hang
- sanitizer가 탐지한 undefined behavior

퍼징의 핵심은 단순히 랜덤 값을 넣는 것이 아니다. 입력을 생성하고, 실행 결과를 관찰하고, 의미 있는 입력을 다시 변형하는 반복 구조가 중요하다.

### 퍼징이 필요한 이유

- 자동화: 수천, 수백만 개의 테스트 케이스를 자동 실행할 수 있다.
- 효율성: 사람이 놓치기 쉬운 edge case를 찾을 수 있다.
- 보안성: 입력 파싱, 파일 처리, 프로토콜 처리 취약점을 찾는 데 강하다.
- 신뢰성: 크래시와 비정상 동작을 줄여 프로그램 안정성을 높인다.

Adobe, Google, Microsoft 같은 기업도 퍼징을 보안 개발 프로세스의 중요한 요소로 사용한다.

## 2. 퍼징의 분류

퍼징은 입력 생성 방식, 대상 프로그램에 대한 정보 수준, 피드백 방식에 따라 나눌 수 있다.

### 입력 생성 방식에 따른 분류

#### Dumb Fuzzing

입력 형식을 거의 고려하지 않고 완전히 무작위에 가까운 값을 넣는 방식이다.

장점:

- 구현이 쉽다.
- 대상 프로그램에 대한 이해가 거의 없어도 시작할 수 있다.

단점:

- 많은 입력이 초기 검증 단계에서 거부된다.
- 깊은 코드 경로까지 도달하기 어렵다.

#### Generation-Based Fuzzing

입력 형식이나 프로토콜 스펙을 기반으로 테스트 케이스를 생성하는 방식이다.

예시:

- PNG, PDF, ZIP 같은 파일 포맷
- HTTP, DNS, TLS 같은 프로토콜
- JSON, XML 같은 구조화 데이터

형식에 맞는 입력을 만들기 때문에 초기 파서를 통과할 가능성이 높다. 대신 입력 스펙을 작성하거나 모델링해야 한다.

#### Mutation-Based Fuzzing

정상 입력(seed)을 조금씩 변형해 새로운 입력을 만드는 방식이다.

예시:

- 일부 바이트 뒤집기
- 문자열 삽입
- 블록 삭제 또는 복제
- 숫자 값을 경계값으로 변경

초기 seed corpus 품질이 중요하다. 좋은 seed가 있으면 빠르게 깊은 경로에 도달할 수 있다.

### 프로그램 구조 인식 여부에 따른 분류

#### Black-Box Fuzzing

대상 프로그램의 내부 구조를 모르는 상태에서 입력과 출력만 보고 테스트한다.

장점:

- 소스 코드가 없어도 가능하다.
- 상용 바이너리나 제3자 프로그램 테스트에 적합하다.

단점:

- 어떤 코드 경로가 실행됐는지 알기 어렵다.
- 비효율적인 입력이 많이 생성될 수 있다.

#### Gray-Box Fuzzing

코드 커버리지 같은 제한적인 실행 정보를 피드백으로 사용하는 방식이다. 현재 가장 널리 쓰이는 방식이다.

대표 도구:

- AFL++
- libFuzzer
- Honggfuzz

새로운 코드 경로를 발견한 입력을 corpus에 저장하고, 그 입력을 다시 변형해 더 많은 경로를 탐색한다.

#### White-Box Fuzzing

소스 코드와 프로그램 구조를 적극적으로 분석해 경로를 탐색하는 방식이다. symbolic execution이나 constraint solving을 결합하기도 한다.

장점:

- 특정 경로를 논리적으로 추적할 수 있다.
- 깊은 조건문을 통과하는 입력을 만들 수 있다.

단점:

- 구현과 분석 비용이 높다.
- 대규모 프로그램에서는 path explosion 문제가 발생한다.

### 피드백 방식에 따른 분류

#### Coverage-Guided Fuzzing

프로그램의 코드 커버리지를 측정하고, 새로운 경로를 실행한 입력을 우선적으로 보존하는 방식이다.

작동 흐름:

1. 초기 seed corpus를 실행한다.
2. 입력을 변형한다.
3. 대상 프로그램에 넣어 실행한다.
4. 새로운 coverage가 나오면 corpus에 저장한다.
5. 저장된 입력을 다시 변형한다.

대표 퍼저는 AFL++, libFuzzer, Honggfuzz다.

#### Feedback-Driven Fuzzing

커버리지뿐 아니라 다양한 실행 피드백을 활용하는 방식이다.

피드백 예시:

- 코드 커버리지
- 실행 경로
- 분기 정보
- CPU 성능 카운터
- Intel BTS, Intel PT 같은 하드웨어 추적 정보

## 3. 주요 퍼저 비교

### AFL++

AFL++는 기존 AFL(American Fuzzy Lop)을 개선한 커뮤니티 기반 퍼저다. 현재도 널리 쓰이는 대표적인 gray-box coverage-guided fuzzer다.

주요 특징:

- 계측 기반 커버리지 추적
- mutation-based fuzzing
- power scheduling 지원
- 병렬 퍼징 지원
- QEMU 모드를 통한 바이너리 퍼징 가능
- 다양한 sanitizer와 연동 가능

기본 사용 예시:

```bash
afl-fuzz -i input_dir -o output_dir -- ./target_program @@
```

병렬 퍼징 예시:

```bash
afl-fuzz -i input_dir -o output_dir -M main -- ./target @@
afl-fuzz -i input_dir -o output_dir -S variant1 -- ./target @@
afl-fuzz -i input_dir -o output_dir -S variant2 -- ./target @@
```

장점:

- 안정적이고 검증된 도구다.
- 설정 옵션이 많다.
- 문서와 커뮤니티가 좋다.
- 소스 코드와 바이너리 대상 모두에 적용할 수 있다.

단점:

- 옵션이 많아 초보자에게 복잡할 수 있다.
- 대상에 맞는 harness 작성이 필요할 때가 많다.

### libFuzzer

libFuzzer는 LLVM 프로젝트에서 제공하는 in-process coverage-guided fuzzer다. 대상 함수를 같은 프로세스 안에서 반복 호출하므로 실행 속도가 빠르다.

주요 특징:

- LLVM SanitizerCoverage 기반 커버리지 측정
- AddressSanitizer, UndefinedBehaviorSanitizer와 연동이 좋다.
- 라이브러리 함수 단위 fuzzing에 적합하다.
- Clang 환경에서 사용하기 쉽다.

fuzz target 예시:

```cpp
#include <cstddef>
#include <cstdint>

extern "C" int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    if (size < 4) {
        return 0;
    }

    if (data[0] == 'H' && data[1] == 'I') {
        // target logic
    }

    return 0;
}
```

컴파일 및 실행:

```bash
clang++ -fsanitize=fuzzer,address fuzz_target.cpp target_lib.cpp -o fuzz_target
./fuzz_target -dict=dict.txt input_corpus/
```

장점:

- 빠르다.
- 사용법이 단순하다.
- sanitizer와 결합하기 좋다.
- 라이브러리 테스트에 적합하다.

단점:

- 바이너리만 있는 대상에는 적합하지 않다.
- in-process 구조라 전역 상태나 리소스 정리가 중요하다.
- 2022년 후반부터는 유지보수 중심 모드다.

### Jackalope

Jackalope는 Google Project Zero에서 개발한 커스터마이징 가능한 coverage-guided binary fuzzer다.

주요 특징:

- TinyInst 기반 동적 계측
- Windows, macOS, Linux, Android 지원
- 바이너리 퍼징에 강점
- grammar-based fuzzing 지원
- 서버/클라이언트 구조를 통한 분산 퍼징 가능

기본 구조:

```text
Fuzzer
  ├── Instrumentation
  ├── Mutators
  ├── Sample Delivery
  └── Coverage Tracking
```

사용 예시:

```bash
./fuzzer -in input_dir -out output_dir -t 1000 -delivery shmem \
  -instrument_module target -target_module target \
  -target_method __Z4fuzzPc -nargs 1 \
  -- ./target @@
```

장점:

- 소스 코드 없이 바이너리 대상으로 사용할 수 있다.
- Windows와 macOS 지원이 좋다.
- 구조가 깔끔해 확장하기 좋다.
- grammar 기반 입력 생성이 가능하다.

단점:

- AFL++나 libFuzzer보다 커뮤니티가 작다.
- 문서와 사례가 상대적으로 적다.
- 고급 mutation 전략은 직접 보강해야 할 수 있다.

### Honggfuzz

Honggfuzz는 Google에서 개발한 보안 중심 feedback-driven fuzzer다. 소프트웨어 커버리지뿐 아니라 하드웨어 기반 피드백도 활용할 수 있다.

주요 특징:

- LLVM SanitizerCoverage 지원
- Intel BTS, Intel PT 기반 하드웨어 피드백 지원
- 멀티 프로세스, 멀티 스레드 실행
- persistent mode 지원
- ptrace 기반 모니터링

사용 예시:

```bash
honggfuzz -i input_dir -- ./target @@
honggfuzz -i input_dir -P -- ./persistent_target
honggfuzz -i input_dir -n 12 -v -- ./target @@
```

하드웨어 피드백 예시:

```bash
honggfuzz -i input_dir --linux_perf_bts_edge -- ./target @@
honggfuzz -i input_dir --linux_perf_ipt_block -- ./target @@
honggfuzz -i input_dir --linux_perf_instr -- ./target @@
```

장점:

- 실행 속도가 빠르다.
- 하드웨어 기반 피드백을 사용할 수 있다.
- crash 감지와 모니터링이 강력하다.
- 여러 OS를 지원한다.

단점:

- 하드웨어 피드백은 Linux Intel 환경 의존성이 있다.
- AFL++보다 자료가 적게 느껴질 수 있다.

## 5. 실전 선택 가이드

### 초급자

처음 시작한다면 libFuzzer가 좋다.

- 설정이 단순하다.
- Clang에 포함되어 있다.
- sanitizer와 함께 쓰기 쉽다.

### 소스 코드가 있는 경우

AFL++ 또는 Honggfuzz를 선택한다.

- AFL++: 안정적이고 설정 옵션이 풍부하다.
- Honggfuzz: 빠르고 다양한 피드백 모드를 제공한다.

### 바이너리 퍼징

Jackalope를 고려한다.

- 소스 코드 없이 동적 계측이 가능하다.
- Windows/macOS 대상에 유리하다.

### 대규모 프로젝트 또는 CI 환경

AFL++ 또는 Honggfuzz가 적합하다.

- 병렬 실행이 좋다.
- 자동화하기 쉽다.
- sanitizer와 결합하기 좋다.

### 복잡한 파일 형식

grammar-based fuzzing을 고려한다.

- 입력 구조를 유지할 수 있다.
- 무작위 변이보다 깊은 파서 경로에 도달하기 쉽다.
- Jackalope 같은 도구를 활용할 수 있다.

## 6. 퍼징 실전 팁

### 좋은 Seed Corpus 준비

Seed corpus는 퍼저가 변형을 시작할 정상 입력 모음이다.

```text
input_dir/
  ├── valid_input1.txt
  ├── valid_input2.bin
  ├── sample_file.pdf
  └── protocol_packet.bin
```

좋은 corpus 기준:

- 유효한 입력이어야 한다.
- 가능한 다양한 구조를 포함해야 한다.
- 너무 큰 파일보다 작고 핵심적인 샘플이 좋다.
- checksum, HMAC 같은 검증 값은 제거하거나 우회하는 편이 좋다.

### Dictionary 활용

Dictionary는 퍼저가 자주 삽입할 토큰 목록이다. 파서나 프로토콜 테스트에서 효과가 크다.

```text
kw1="if"
kw2="while"
kw3="malloc"
kw4="NULL"
"\x00"
"\xFF"
"\xDE\xAD\xBE\xEF"
```

AFL++에서 dictionary를 사용할 때:

```bash
afl-fuzz -i in -o out -x dict.txt -- ./target @@
```

### Sanitizer와 함께 사용

Sanitizer는 프로그램 실행 중에 메모리 오류, 정의되지 않은 동작, 정수 오버플로 같은 문제를 감지하도록 컴파일 단계에서 검사 코드를 삽입하는 도구다.

퍼저는 이상한 입력을 많이 넣어 crash를 유도하고, sanitizer는 그 crash의 원인을 더 정확하게 보여준다. 즉, 퍼저가 입력을 찾고 sanitizer가 버그의 종류와 위치를 설명해주는 구조다.

대표적인 sanitizer는 다음과 같다.

| Sanitizer | 탐지 대상 |
|---|---|
| AddressSanitizer(ASan) | heap/stack buffer overflow, use-after-free |
| UndefinedBehaviorSanitizer(UBSan) | integer overflow, null pointer dereference, 잘못된 shift 같은 undefined behavior |
| MemorySanitizer(MSan) | 초기화되지 않은 메모리 사용 |
| LeakSanitizer(LSan) | 메모리 누수 |
| ThreadSanitizer(TSan) | data race 같은 스레드 동시성 오류 |

퍼징에서는 특히 ASan과 UBSan을 자주 사용한다. 메모리 corruption 계열 취약점은 일반 실행에서는 조용히 지나가거나 나중에 엉뚱한 위치에서 crash가 날 수 있는데, ASan을 켜면 문제가 발생한 지점에 가까운 로그를 얻을 수 있다.

예를 들어 heap buffer overflow가 발생하면 sanitizer 로그에는 대략 다음 정보가 나온다.

- 오류 종류
- 문제가 발생한 주소
- 읽기/쓰기 여부
- stack trace
- 할당된 메모리 영역 정보

이 정보가 있어야 단순 crash를 실제 취약점 후보로 분류하고, root cause를 빠르게 찾을 수 있다.

AFL++ 예시:

```bash
export AFL_USE_ASAN=1
export AFL_USE_UBSAN=1
afl-cc -o target target.c
afl-fuzz -i in -o out -- ./target @@
```

libFuzzer 예시:

```bash
clang++ -fsanitize=fuzzer,address,undefined fuzz_target.cpp target.cpp -o fuzz_target
```

주의할 점도 있다.

- sanitizer를 켜면 실행 속도가 느려지고 메모리 사용량이 늘어난다.
- ASan은 보통 실제 운영 바이너리보다 더 빨리 crash를 발생시킨다.
- MSan은 모든 의존성도 MSan으로 빌드해야 해서 설정이 까다롭다.
- TSan은 성능 비용이 크기 때문에 일반적인 fuzzing 기본값으로 쓰기에는 무겁다.

실전에서는 먼저 ASan, UBSan 조합으로 돌리고, 필요할 때 LSan, MSan, TSan을 따로 적용하는 방식이 현실적이다.

### Crash 분석

퍼저가 crash를 찾으면 바로 취약점이라고 단정하면 안 된다. 재현 가능성, 영향도, root cause를 확인해야 한다.

확인할 항목:

- 같은 입력으로 반복 재현되는가?
- crash 위치가 입력에 의해 제어되는가?
- sanitizer 로그가 어떤 오류를 가리키는가?
- 보안 영향이 있는가?

AFL++ crash 재현 예시:

```bash
cat output/crashes/id:000000,sig:11,src:000001 | ./target
```

GDB 분석 예시:

```bash
gdb ./target
(gdb) run < crash_file
```

## 7. 핵심 정리

퍼징은 입력 처리 버그를 자동으로 찾는 강력한 보안 테스트 기법이다. 개념은 단순하지만, 좋은 결과를 얻으려면 대상 선정, harness 작성, seed corpus 구성, sanitizer 연동, crash 분석까지 함께 이해해야 한다.

퍼저 선택 기준은 다음과 같이 잡으면 된다.

| 상황 | 추천 |
|---|---|
| 처음 배우는 단계 | libFuzzer |
| 소스 코드가 있는 일반 프로젝트 | AFL++ |
| 빠른 반복 실행이 중요한 경우 | libFuzzer, Honggfuzz |
| 바이너리 대상 | Jackalope, AFL++ QEMU 모드 |
| 대규모 자동화 | AFL++, Honggfuzz |
| 복잡한 파일 포맷 | Grammar-based fuzzing |

효과적인 퍼징의 핵심은 세 가지다.

1. 좋은 seed corpus를 준비한다.
2. sanitizer와 coverage feedback을 적극 활용한다.
3. crash를 재현하고 원인을 분석한다.

## 9. 용어 정리

| 용어 | 설명 |
|---|---|
| Corpus | 퍼저가 사용하는 테스트 케이스 모음 |
| Seed | 초기 입력값 |
| Mutation | 기존 입력을 변형해 새 입력을 만드는 작업 |
| Coverage | 실행된 코드 범위 |
| Crash | 프로그램 비정상 종료 |
| Hang | 프로그램이 멈추거나 제한 시간 안에 종료되지 않는 상태 |
| Instrumentation | 실행 정보를 얻기 위해 프로그램에 계측 코드를 삽입하는 작업 |
| Black-box | 내부 구조를 모르는 상태에서 테스트하는 방식 |
| Gray-box | 일부 실행 정보만 활용하는 방식 |
| White-box | 소스 코드와 구조 정보를 적극적으로 활용하는 방식 |
