# Fuzzing과 Fuzzer

 취약점 분석의 핵심 자동화 기법

## 퍼징(Fuzzing)이란 무엇인가?

`퍼징(Fuzz Testing)`은 소프트웨어의 보안 취약점을 발견하기 위해 사용되는 자동화된 테스트 기법이다. 프로그램에 비정상적, 예상치 못한, 혹은 무작위 데이터(Fuzz)를 입력값으로 주입하고, 
그 과정에서 발생하는 크래시, 코드 단언 오류(Assertion failure), 메모리 누수 등을 모니터링한다.

단순히 무작위 값을 넣는 것이 아니라, 프로그램의 정상 동작 여부를 판별하는 `oracle`이 핵심이다

보안 연구자에게 퍼징은 수동 분석으로 찾기 힘든 복잡한 버그를 대규모로 찾아낼 수 있게 해주는 필수적인 도구이다.

fuzzing = 테스트 방법

fuzzer = 그걸 실행하는 프로그램

## 퍼저(Fuzzer)의 분류

퍼징은 크게 대상 프로그램에 대한 이해도와 데이터 생성 방식에 따라 나뉜다.

### 대상 이해도에 따른 분류 (Knowledge-based)

`Black-box Fuzzing`: 프로그램 내부 구조를 모르는 상태에서 입력/출력만 확인한다다. 구현이 쉽지만 효율이 떨어질 수 있다. (네트워크 프로토콜이나 폐쇠형 소프트웨어 테스트에 적합) 
<br>내부 경로를 모르기 때문에 효율이 낮고, `복잡한 로직을 통과하기 어렵다` !!!!!!!

`White-box Fuzzing`: 프로그램 소스 코드를 분석하여 모든 실행 경로를 체계적으로 탐색한다다. 분석 비용이 높다. 매우 정교한 버그를 찾을 수 있지만<br> `계산 복잡도가 너무 높아 대규모 프로그램에는 적용하기 어렵다`!!!!!!

`Grey-box Fuzzing`: 가장 대중적인 방식으로, 소스 코드 전체를 분석하지는 않지만 컴파일 시점이나 실행 시점에 코드에 센서를 심어 실행경로를 추적한다. 현대 취약점 분석의 주류 (예: AFL++)

### 입력 생성 방식에 따른 분류 (Input Generation)

`Mutation-based` (변이 기반): 정상적인 입력값(Seed)을 바탕으로 비트를 뒤집거나 값을 수정하여 변종을 만든다.

`Generation-based` (생성 기반): 특정 파일 형식이나 네트워크 프로토콜의 규칙(RFC 등)을 학습하여 규격에 맞는(그러나 악의적인) 데이터를 생성한다.



## 대표적인 퍼징 도구 분석

| 도구명 | 개발/관리 | 주요 특징 |
| :--- | :--- | :--- |
| AFL++ | Community | 기존 AFL의 개선판으로, 가장 강력한 Grey-box 퍼저이다. 높은 성능과 다양한 변이 알고리즘을 지원한다. |
| Jackalope | Google Project Zero | 소스 코드가 없는 블랙박스 환경(주로 Windows/macOS 바이너리)을 위해 설계된 커스텀 퍼저이다. |
| ClusterFuzz | Google | 구글 인프라에서 수만 대의 머신을 활용해 대규모 퍼징을 수행할 수 있도록 돕는 프레임워크이다. |
| LibFuzzer | LLVM Project | in-process 퍼저, 빠른 execution과 sanitizer와의 강력한 통합이 특징 |

### 하네스가 뭡니까

Fuzzer는 보통

- 입력을 받아서 빠르게 실행하고 크래시/버그를 탐지

하는데 실제 프로그램은 네트워크, 복잡한 초기화 이런게 많아서 

그대로 퍼징하기 어렵대요 

그래서? 핵심 로직만 분리해서 호출하는 하네스가 필요하답니다 !

```c
#include <stdio.h>
#include <stdlib.h>

// 타겟 함수의 선언을 가져온다.
extern void target_function(char *data);

int main(int argc, char *argv[]) {
    if (argc < 2) return 1;

    char buf[100];
    FILE *fp = fopen(argv[1], "rb");
    if (fp) {
        size_t len = fread(buf, 1, 99, fp);
        buf[len] = '\0';
        fclose(fp);
    }

    // 하네스가 타겟 함수를 호출하여 데이터를 주입함
    target_function(buf);

    return 0;
}

// AFL++ 하네스 
```

AFL은 보통 파일 입력 기반

stdin이나 파일을 통해 데이터를 받는다.

이 외에도 stdin 입력, persistent mode 등 모드나 빠르게 하는 다른 방식들이 있다.

```bash
mkdir in out
echo "hi" > in/seed.txt
```

in/ = inputDirectory : 퍼저가 변이을 시작할 기초 데이터인 seed를 넣어둔다. 시드는 작고 효율적일수록 좋다.

out/ = outputDirectory : 퍼저가 발견한 흥미로운 입력값(queue)과 프로그램을 터트린 데이터(crashes)를 저장하는 공간.

```bash
afl-fuzz -i in -o out -- ./fuzzer_target @@

afl-fuzz = AFL++의 메인 엔진 실행
-i in = 시드 파일이 있는 입력 디렉토리 지정
-o out = 결과를 저장할 출력 디렉토리 지정
-- ./test_fuzz = --뒤에 퍼징할 대상 프로그램과 인자를 적는다. 컴파일한 파일이 타겟

소스파일이 없는 라이브러리 = alf-fuzz -Q (QEMU 모드)
```

AFL++의 중요한 실행 명령

```bash
파일 입력:
afl-fuzz -i seeds -o findings -- ./harness @@

stdin 입력:
afl-fuzz -i seeds -o findings -- ./harness

dictionary 사용:
afl-fuzz -i seeds -o findings -x dict.txt -- ./harness @@

timeout 지정:
afl-fuzz -i seeds -o findings -t 1000 -- ./harness @@

메모리 제한 해제:
afl-fuzz -i seeds -o findings -m none -- ./harness @@
```

## 퍼징 프로세스

1. `대상 선정` (Targeting): 공격 표면(Attack Surface)이 넓은 라이브러리나 실행 파일을 선정한다.
2. `시드 준비` (Seed Corpus): 퍼저가 변이를 시작할 수 있는 양질의 정상 입력 파일들을 모은다.
3. `퍼징 실행` (Execution): 퍼저를 가동하여 반복적인 입력 주입을 시작한다.
4. `크래시 분석` (Analysis): 발견된 크래시가 실제 보안 취약점(CVE)으로 이어질 수 있는지 분석한다.

## 실제 버그바운티 워크플로우

실제 보안 연구원들이 기업의 취약점을 찾고 보상금을 받는 과정은 다음과 같은 체계적인 단계를 거친다.

| 단계 | 수행 활동 | 세부 내용 |
| :--- | :--- | :--- |
| 1. 타겟 선정 | 분석 대상 식별 | 오픈소스 라이브러리(libpng, OpenSSL 등)나 기업 자산 중 공격 표면(Attack Surface)이 넓은 곳을 선정. |
| 2. 대규모 퍼징 | 고성능 인프라 가동 | 개인 PC가 아닌 클라우드 서버(GCP, AWS) 수십 대를 동원하여 AFL++를 24시간 내내 무한 반복 실행. |
| 3. Crash 분석 | 원인 파악 및 검증 | `unique crashes` 발생 시, GDB와 같은 디버거를 이용해 메모리 어느 지점에서 왜 터졌는지(BOF, UAF 등) 정밀 분석. |
| 4. 보고(Report) | 취약점 제보 | 발견된 취약점의 위험도와 함께 크래시를 유발하는 파일(PoC)을 기업에 전달하고 패치를 권고. |
