## 논문을 고른 이유

IoT 펌웨어 보안은 최근 매우 중요한 분야이며, 특히 정적 분석(taint analysis)은 취약점을 많이 찾을 수 있지만 실제 취약점인지 검증하기 어렵다는 문제가 있다. 

기존에 취약점 분석을 공부하면서 탐지보다 검증이 더 어렵다는 점을 느꼈기 때문에, 이를 자동화하려는 접근이 흥미로워 이 논문을 선택하였다.

그리고 보안 전문가로 스펙트럼을 넓히기 위해 레드팀 활동에 관심을 가지고 있지만, 기존의 지식이 웹 보안에 편중되어 있다는 한계를 느꼈다. 이 논문은 이러한 기술적 공백을 채우고, 실제 소프트웨어의 메모리 취약점을 탐지하는 메커니즘을 이해하는 데 가장 적합할 것이라 판단했다. 특히 대규모 버그바운티에서 활용되는 자동화 기법을 학습하고 싶어 이 논문을 선정했다.

## 문제 정의

기존 `taint analysis`는 다음과 같은 문제를 가진다.

- 많은 취약점 후보를 생성하지만 `false positive`가 많음
- `entry point` 및 `parameter` 정보를 제공하지 않음
- 실제 취약점 재현(PoC 생성)이 어려움
- 검증 과정이 수작업에 의존 → 비효율적

따라서 논문에서는 taint analysis 결과를 자동으로 검증하는 문제를 해결하고자 한다.

### taint analysis란?

taint analysis는 외부 입력(사용자 입력 등)이 프로그램 내부에서 어떻게 전파되어 <br>
위험한 함수(sink)에 도달하는지를 추적하는 정적 분석 기법이다.

예를 들어 사용자 입력이 검증 없이 strcpy와 같은 함수로 전달된다면, <br>
버퍼 오버플로우와 같은 취약점이 발생할 수 있다.

### false positive가 왜 문제인가?

taint analysis는 많은 취약점 후보를 찾을 수 있다는 장점이 있지만, <br>
실제로는 실행되지 않는 경로나 이미 필터링된 입력까지 취약점으로 판단하는 경우가 많다.

이러한 false positive가 많아지면 분석가는
각 결과를 수작업으로 검증해야 하므로 시간과 비용이 크게 증가한다.

## 핵심 아이디어

`Bond`의 핵심 아이디어는 다음과 같다.

> constraint 정보를 활용하여 fuzzing을 방향성 있게 수행한다

기존 `fuzzing`:

랜덤 입력 → 비효율적

`Bond`:

- taint report 기반
- constraint 추출
- constraint 기반 mutation
→ 취약점 경로를 직접 공략

### constraint ..?

프로그램이 특정 취약점 경로에 도달하기 위해 입력값이 반드시 만족해야 하는 조건을 의미한다.

일반 퍼징은 미로에서 눈을 가리고 벽에 부딪히면서 길을 찾는 것과 같다.

하지만 프로그램 깊숙한 곳에 있는 취약점을 찾으려면 복잡한 조건문을 통과해야 한다.

Constraint 기반 방향성 퍼징은 목표 지점까지 가기 위한 수학적 제약 조건을 미리 풀어서

정답지를 들고 효율적으로 공격 경로를 찾아가는 퍼징 기법이라고 이해하면 된다.

## 방법론 / 시스템 설명

![alt text](/assets/images/paper1/image-1.png)

`Bond`는 다음 4단계로 구성된다.

### Entry Point Identification (진입점 식별)

- CFG / Call Graph 분석
- 숨겨진 dispatch 구조 탐색
- 실제 사용자 입력이 들어오는 entry point 복원

IoT에서는 API가 명확하지 않기 때문에 매우 중요

이 과정이 중요한 이유는 IoT firmware의 경우 일반적인 프로그램처럼 명확한 API나 main 함수가 존재하지 않고, <br>
여러 dispatch 로직을 통해 간접적으로 함수가 호출되기 때문이다.

따라서 정확한 entry point를 찾지 못하면 취약점 경로 자체에 도달할 수 없다.

### Reachable Region Analysis (도달 가능 영역 분석)

- entry → sink까지 도달 가능한 코드 영역 계산
- 사용자 입력이 영향을 주는 parameter 추출

이 단계는 전체 코드가 아니라 취약점과 관련된 경로만을 분석 대상으로 제한하기 위한 과정이다.

이를 통해 불필요한 코드 영역을 제거하고, fuzzing이 집중해야 할 경로를 좁힐 수 있다.

### Constraint Analysis (제약 조건 분석)

입력값이 만족해야 하는 조건을 분석

이 단계에서 추출된 constraint는 이후 fuzzing 단계에서 입력 생성의 기준으로 사용된다.

즉, constraint 분석 결과가 정확할수록 취약점이 존재하는 경로를 더 빠르게 탐색할 수 있다.

#### Constraint 종류

- Mandatory constraint (반드시 만족)
해당 조건을 만족하지 않으면 취약점 경로 자체에 도달할 수 없는 필수 조건

- Partial constraint (경로 선택)
여러 경로 중 하나를 선택하게 만드는 조건으로, 취약점 탐색 범위를 넓히는 데 사용된다.

- Non-constraint (영향 없음)
경로 결정에 영향을 주지 않는 입력으로, 우선순위가 가장 낮게 설정된다.

#### Semantic 유형 (6가지)

- 문자열 비교 (strcmp)
- 숫자 범위 (atoi)
- 바이트 조건
- null 조건
- 네트워크 포맷 (IP 등)
- heuristic 조건



### Directed Black-box Fuzzing (방향성 블랙박스 퍼징)

LLM을 활용하는 이유는 IoT 기기의 HTTP 요청 구조가 공식 문서 없이 비공개로 구현된 경우가 많기 때문이다.

LLM은 기존 PoC나 공개된 요청 패턴을 기반으로 올바른 요청 형식을 추론할 수 있으며, <br> 
이를 통해 fuzzing이 실제로 처리 가능한 입력을 생성하도록 돕는다.

![alt text](/assets/images/paper1/image-2.png)

- LLM 기반 HTTP 요청 템플릿 생성
- constraint 기반 mutation 우선순위 적용

이 단계에서는 앞서 추출한 constraint를 활용하여 입력을 생성하고, 실제로 취약점이 발생하는지를 검증한다.

특히 mandatory constraint를 먼저 만족시키고,<br>
그 이후에 partial constraint를 탐색하는 방식으로 효율적인 경로 탐색이 가능해진다.

#### 우선순위

1. Mandatory constraint
2. Partial constraint
3. Non-constraint

핵심: 중요한 입력부터 fuzzing

## 평가 및 결과 해석

| 항목 | 결과 |
|------|------|
| 총 taint report | 2,776 |
| 검증 성공 | 1,349 |
| 성공률 | 48.6% |
| Known CVE 재현율 | 91.67% |
| 0-day 발견 | 155 |

### 전체 성능

- 2,776개 taint report 분석
- 1,349개 실제 취약점 검증 성공
→ 약 48.6% 성공률

### 기존 기법과 비교

- 최대 5.5배 성능 향상
- 기존 fuzzing 대비 훨씬 효율적

### Known Vulnerability 재현

- 60개 CVE 중
→ 91.67% 재현 성공

### 추가 성과

- 155개 0-day 취약점 발견
- 108개 CVE 할당

## 한계점

1. taint analysis 품질에 의존
2. cross-process 취약점 처리 불가
3. HTTP 중심 → binary protocol 확장 어려움
4. 상태 의존 취약점 탐지 제한
5. LLM 기반 템플릿 생성 → 완전 신뢰 어려움
6. 실제 취약점 트리거는 가능하지만 exploit까지 자동 생성하지는 못함

### IoT firmware에서 fuzzing이 어려운 이유

IoT 환경에서는 일반 소프트웨어와 달리 실행 환경이 제한적이고, 디버깅이나 instrumentation과 같은 동적 분석 기법을 적용하기 어렵다.

또한 입력 형식이 제조사 마다 다르고, 정확한 프로토콜을 맞추지 않으면 요청이 아예 처리되지 않는다.

이 때문에 단순한 랜덤 fuzzing은 대부분의 입력이 초기 단계에서 필터링되어 비효율적이다.

## 기존 연구와의 차별점 

| 방법 | 특징 | 한계 |
|------|------|------|
| Black-box fuzzing | 랜덤 입력 기반 | 비효율적, 경로 도달 어려움 |
| Greybox fuzzing | 실행 정보 활용 | IoT에서 적용 어려움 |
| Symbolic execution | 정확한 경로 탐색 | path explosion |
| Bond (제안) | constraint 기반 fuzzing | taint report 품질 의존 |

 Bond의 차별점:

- constraint 기반 방향성 fuzzing
- 정적 분석 + 동적 검증 결합
- IoT 환경에 최적화

## 느낀 점

```bash
노말틱 유튜브에서 ai가 찾아내는 취약점이 허구가 많다는 것을 봤는데 
이 논문은 단순히 취약점을 찾는 것이 아니라  
찾은 결과를 실제 취약점으로 검증하는 문제를 해결했다는 점이 인상적이었다.

특히 인상 깊었던 점:

- constraint 기반 접근
- fuzzing 효율 개선
- LLM 활용

보안 실무에서도 매우 유용할 것이라고 생각했다.

그리고 이 연구는 발견 → 검증 이라는 보안 분석 흐름에서
검증 단계의 중요성을 강조했다는 점에서 의미가 있다고 생각한다.

레드팀과 버그바운티에 관심이 있었지만, 바이너리나 시스템 쪽은 막연한 두려움 때문에 선뜻 도전하지 못했다. 하지만 이번 실습을 해보니, 생각보다 흥미로웠다. 이제 분야를 가리지 않고 도전하는 잡식성으로 바뀌어야 겠다.
```
