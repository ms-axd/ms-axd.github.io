## 처음 든 생각

영상을 보면서 가장 먼저 느낀 점은 스마트 컨트랙트가 그냥 서버 코드와 다르다는 것이다.

일반 웹 서비스에서 버그가 나면 서버를 고치고 다시 배포하면 된다.

하지만 스마트 컨트랙트는 블록체인 위에 올라가고, 돈이나 토큰 같은 자산을 직접 다룬다.

그래서 코드 하나가 잘못되면 단순 오류가 아니라 실제 금전 피해로 이어질 수 있다.

영상의 핵심도 이 부분이었다.

```text
블록체인이 안전해도
그 위에 올라간 스마트 컨트랙트 코드가 취약하면
자산은 탈취될 수 있다.
```

DAO 해킹도 이 관점에서 설명된다.

이더리움 자체가 뚫린 것이 아니라, 이더리움 위에서 실행되던 스마트 컨트랙트의 로직에 문제가 있었던 것이다.

## DAO 해킹과 재진입 공격

영상에서 가장 중요하게 다룬 내용은 `reentrancy`다.

재진입 공격이라고 한다.

처음 들었을 때는 이름이 어렵게 느껴졌는데, 구조는 생각보다 단순했다.

문제는 출금 순서다.

```text
1. 사용자에게 Ether를 보낸다.
2. 사용자 잔액을 줄인다.
```

이렇게 처리하면 Ether를 보내는 과정에서 공격자가 다시 출금 함수를 호출할 수 있다.

아직 잔액이 줄어들기 전이기 때문에 같은 잔액으로 여러 번 출금할 수 있다.

DAO 해킹도 이런 식의 재진입 문제가 핵심이었다고 설명한다.

이 부분을 보고 스마트 컨트랙트에서는 코드 실행 순서가 정말 중요하다는 걸 알게 됐다.

출금 로직은 다음처럼 되어야 한다.

```text
1. 조건을 확인한다.
2. 내부 상태를 먼저 바꾼다.
3. 마지막에 외부로 Ether를 보낸다.
```

정리하면 돈을 보내기 전에 내부 잔액부터 바꿔야 한다.

## 접근 제어 문제

영상에서는 Parity Wallet 사례도 나온다.

여기서 중요하게 본 부분은 함수 접근 권한이다.

Solidity에서는 함수에 `public`, `private`, `internal` 같은 접근 범위를 지정할 수 있다.

문제는 원래 관리자만 호출해야 하는 함수가 외부에서도 호출 가능하게 열려 있을 때 생긴다.

예를 들면 이런 흐름이다.

```text
관리자 설정 함수가 public
-> 공격자가 직접 호출
-> 소유권 탈취
-> 자산 이동
```

Web3를 처음 배우는 입장에서는 `public`이라는 키워드가 단순히 공개 함수라는 뜻으로만 보였다.

그런데 스마트 컨트랙트에서는 이게 곧 자산과 권한 문제로 이어질 수 있다는 점이 중요했다.

앞으로 컨트랙트 코드를 볼 때는 먼저 이 질문을 해야겠다고 느꼈다.

```text
이 함수는 누가 호출할 수 있는가?
이 함수가 호출되면 어떤 상태가 바뀌는가?
```

## send 반환값 확인

영상에서는 Ether를 보내는 코드에서 반환값을 확인하지 않는 문제도 설명한다.

`send` 같은 함수는 실패할 수 있다.

그런데 실패했는지 확인하지 않고 다음 로직을 실행하면 이상한 상태가 생긴다.

```text
Ether 전송 실패
-> 실패 여부 확인 안 함
-> 내부 상태는 지급 완료로 처리
-> 실제 자산 상태와 코드 상태가 달라짐
```

처음에는 돈을 보내는 함수면 당연히 성공하거나 실패하면 멈출 거라고 생각했다.

하지만 영상에서는 이런 외부 호출의 결과를 직접 확인해야 한다는 점을 강조한다.

스마트 컨트랙트에서는 “됐겠지”라고 가정하면 안 된다.

---

## 오버플로와 언더플로

영상에서는 정수 오버플로와 언더플로도 나온다.

스마트 컨트랙트에서는 잔액, 토큰 수량, 보상 금액처럼 숫자 계산이 많다.

그래서 숫자 계산 오류가 바로 자산 문제로 이어질 수 있다.

오버플로는 숫자가 최대 범위를 넘어가면서 이상한 값이 되는 문제다.

언더플로는 0보다 작은 값이 되면서 이상한 값이 되는 문제다.

예를 들어 잔액이 `0`인데 `1`을 빼면, 제대로 막지 않을 경우 아주 큰 숫자로 바뀔 수 있다.

그러면 실제로는 토큰이 없는데 많은 토큰을 가진 것처럼 처리될 수 있다.

```text
잔액 계산 오류
-> 잘못된 토큰 수량 발생
-> 전송 또는 인출 가능
```

이 부분은 Web3에서 숫자 하나도 그냥 계산하면 안 된다는 느낌을 줬다.

## 가스 한계와 DoS

블록체인에서는 코드를 실행할 때 가스가 필요하다.

영상에서는 이 가스 한계 때문에 서비스 거부 공격이 가능할 수 있다고 설명한다.

예를 들어 모든 사용자 목록을 반복문으로 순회하는 함수가 있다고 가정한다.

처음에는 사용자가 적어서 괜찮다.

하지만 사용자가 많아지면 반복문 실행 비용이 커지고, 결국 가스 한계를 넘을 수 있다.

```text
사용자 수 증가
-> 반복문 비용 증가
-> 가스 한계 초과
-> 함수 실행 실패
```

이걸 보면서 스마트 컨트랙트에서는 기능이 논리적으로 맞아도 실행 비용 때문에 문제가 될 수 있다는 걸 알게 됐다.

Web3에서는 성능 문제가 보안 문제로도 이어질 수 있다.

---

## 온체인 데이터는 숨겨지지 않는다

영상에서 인상 깊었던 내용 중 하나는 `private`에 대한 설명이다.

Solidity에서 `private`이라고 적으면 외부에서 바로 호출하지 못하게 막을 수 있다.

하지만 블록체인에 저장된 데이터 자체가 비밀이 되는 것은 아니다.

온체인에 올라간 데이터는 결국 읽을 수 있다고 봐야 한다.

그래서 이런 값은 그대로 넣으면 위험하다.

| 데이터 | 위험한 이유 |
| --- | --- |
| 비밀번호 | 블록체인 저장소를 분석하면 노출될 수 있음 |
| 랜덤 값 | 예측되거나 읽힐 수 있음 |
| 게임 정답 | 참여자가 미리 볼 수 있음 |
| 입찰 정보 | 공개되면 로직이 깨질 수 있음 |

처음에는 `private`이면 진짜 숨겨지는 줄 알았다.

하지만 영상에서는 `private`은 접근 제한이지 암호화가 아니라는 점을 보여준다.

```text
블록체인에 올린 데이터는 숨겨진 데이터가 아니다.
```

## 도구와 코드 리뷰

영상에서는 `solgraph`, `Oyente` 같은 도구도 소개한다.

이런 도구는 컨트랙트 구조를 보거나 취약한 패턴을 찾는 데 도움을 준다.

하지만 발표자는 도구만 믿으면 안 된다는 흐름으로 설명한다.

도구가 모든 문제를 자동으로 찾아주는 것은 아니다.

특히 컨트랙트의 경제 구조나 비즈니스 로직은 사람이 직접 이해해야 한다.

예를 들어 이런 질문은 도구만으로 판단하기 어렵다.

```text
이 보상 구조를 악용할 수 있는가?
누가 관리자 권한을 가져야 하는가?
사용자가 특정 순서로 행동하면 이득을 볼 수 있는가?
```

스마트 컨트랙트 보안은 단순히 문법 오류를 찾는 일이 아니라는 점을 알 수 있었다.

## 정리

이 영상을 보고 스마트 컨트랙트 보안이 왜 중요한지 조금 이해됐다.

처음에는 블록체인이 안전하면 그 위의 서비스도 안전할 거라고 막연히 생각했다.

이 영상의 핵심은 그게 전혀 아니었고 블록체인과 다르게 서비스 자체도 안전해야 한다 였다.
Web3 입문자 입장에서 기억할 내용은
```text
돈을 보내기 전에 내부 상태를 먼저 바꿔야 한다.
외부에서 호출 가능한 함수는 항상 조심해야 한다.
send 같은 외부 호출 결과를 확인해야 한다.
숫자 계산 오류가 자산 탈취로 이어질 수 있다.
온체인 데이터는 숨겨지지 않는다.
도구보다 컨트랙트 로직 이해가 먼저다.
```

현재 기준으로 보면 영상에 나온 일부 내용은 오래된 부분도 있다.

예를 들어 Solidity는 시간이 지나면서 더 안전한 방향으로 바뀌었고, 스마트 컨트랙트 분석 도구도 많이 발전했다.

하지만 영상에서 말하는 핵심 문제는 아직도 중요하다.

재진입 공격, 접근 제어 실수, 외부 호출 처리, 온체인 데이터 노출 같은 문제는 지금도 Web3 보안에서 계속 나오는 주제다.

```text
예전 영상이지만,
스마트 컨트랙트가 왜 위험한지 이해하는 데는 도움이 된다.
Web3를 처음 공부하는 입장에서 보기에는 충분히 의미 있는 자료였다.
```
