이번에는 `EIP-721`을 보자.

EIP-721은 흔히 `ERC-721`이라고 부르는 NFT 표준이다.

---

## EIP-721이란?

`EIP-721`은 이더리움에서 `Non-Fungible Token`을 표현하기 위한 표준이다.

`Non-Fungible`은 대체 불가능하다는 뜻이다.
즉 각 토큰이 서로 같은 것이 아니라, 고유한 식별자와 의미를 가진다.

ERC-20 토큰은 1개와 다른 1개가 같은 가치로 취급된다.
반면 ERC-721 토큰은 각각의 토큰이 서로 다르다.

예를 들어 다음과 같다.

| 표준 | 성격 | 예시 |
| --- | --- | --- |
| ERC-20 | 대체 가능 토큰 | USDT, UNI, LINK |
| ERC-721 | 대체 불가능 토큰 | NFT 이미지, 게임 아이템, 멤버십, 티켓 |

ERC-20에서 `1 USDT`는 다른 `1 USDT`와 구분할 필요가 없다.
하지만 ERC-721에서 `토큰 ID 1번`과 `토큰 ID 2번`은 서로 다른 자산이다.

---

## 왜 등장했나

ERC-20은 대체 가능한 자산을 표현하기에는 좋다.
하지만 고유한 자산을 표현하기에는 부족하다.

예를 들어 게임 아이템을 생각해보자.

```text
전설 검 #1001
전설 검 #1002
전설 검 #1003
```

이 셋은 이름이 비슷해도 완전히 같은 자산이 아니다.
각각 강화 수치, 소유 이력, 희귀도, 이미지, 메타데이터가 다를 수 있다.

ERC-20으로 이런 자산을 표현하면 문제가 생긴다.
ERC-20은 잔액 중심 모델이기 때문이다.

```text
alice: sword 3개 보유
bob: sword 1개 보유
```

이 방식으로는 Alice가 가진 3개의 sword 중 어떤 sword가 누구에게 이동했는지 명확히 추적하기 어렵다.
각 토큰의 개별 정체성을 표현하기 힘들다.

EIP-721은 이 문제에서 출발했다.

```text
잔액만 보는 토큰이 아니라,
각 토큰 ID의 소유자를 추적할 수 있는 표준이 필요하다.
```

---

## EIP-721이 해결한 문제

EIP-721이 해결한 핵심 문제는 `고유한 디지털 자산을 표준 방식으로 표현하는 것`이다.

### 1. 개별 토큰 식별 문제

ERC-721은 모든 토큰에 `tokenId`를 부여한다.

```text
tokenId 1
tokenId 2
tokenId 3
```

이 `tokenId`가 각 NFT의 고유 식별자 역할을 한다.
그래서 컨트랙트는 특정 토큰이 누구의 것인지 명확히 추적할 수 있다.

### 2. 소유권 추적 문제

ERC-721은 `ownerOf(tokenId)`를 통해 특정 토큰의 소유자를 조회한다.

ERC-20은 주소별 잔액을 중심으로 본다.
ERC-721은 토큰 ID별 소유자를 중심으로 본다.

이 차이가 중요하다.

| 질문 | ERC-20 | ERC-721 |
| --- | --- | --- |
| Alice가 몇 개 가지고 있는가 | `balanceOf(alice)` | `balanceOf(alice)` |
| 특정 토큰은 누구 것인가 | 의미가 약함 | `ownerOf(tokenId)` |
| 토큰마다 개별 메타데이터가 있는가 | 기본 구조에 없음 | `tokenURI(tokenId)`로 표현 가능 |

### 3. NFT 생태계 호환성 문제

표준이 없다면 NFT 마켓플레이스, 지갑, 게임, 블록 익스플로러는 NFT 컨트랙트마다 다른 방식으로 연동해야 한다.

ERC-721은 이 문제를 공통 인터페이스로 해결했다.

지갑은 ERC-721 인터페이스를 지원하면 사용자가 가진 NFT를 표시할 수 있다.
마켓플레이스는 ERC-721 전송 규칙을 기준으로 거래를 처리할 수 있다.
게임은 ERC-721 토큰을 아이템이나 캐릭터 소유권으로 사용할 수 있다.

---

## ERC-721 핵심 함수

ERC-721의 핵심 함수는 다음과 같다.

| 함수 | 설명 |
| --- | --- |
| `balanceOf(address)` | 특정 주소가 가진 NFT 개수 조회 |
| `ownerOf(uint256)` | 특정 tokenId의 소유자 조회 |
| `safeTransferFrom(address,address,uint256)` | NFT 안전 전송 |
| `transferFrom(address,address,uint256)` | NFT 전송 |
| `approve(address,uint256)` | 특정 tokenId 전송 권한 승인 |
| `getApproved(uint256)` | 특정 tokenId에 승인된 주소 조회 |
| `setApprovalForAll(address,bool)` | 특정 operator에게 전체 NFT 관리 권한 부여 |
| `isApprovedForAll(address,address)` | 전체 권한 승인 여부 조회 |

대략적인 인터페이스는 다음과 같다.

```solidity
interface IERC721 {
    function balanceOf(address owner) external view returns (uint256 balance);
    function ownerOf(uint256 tokenId) external view returns (address owner);

    function safeTransferFrom(address from, address to, uint256 tokenId) external;
    function transferFrom(address from, address to, uint256 tokenId) external;

    function approve(address to, uint256 tokenId) external;
    function getApproved(uint256 tokenId) external view returns (address operator);

    function setApprovalForAll(address operator, bool approved) external;
    function isApprovedForAll(address owner, address operator) external view returns (bool);

    event Transfer(address indexed from, address indexed to, uint256 indexed tokenId);
    event Approval(address indexed owner, address indexed approved, uint256 indexed tokenId);
    event ApprovalForAll(address indexed owner, address indexed operator, bool approved);
}
```

## 주요 함수 분석

### ownerOf

`ownerOf`는 특정 NFT의 소유자를 확인한다.

```solidity
ownerOf(1001);
```

이 함수는 `tokenId 1001`의 현재 소유자를 반환한다.
NFT에서 가장 중요한 함수 중 하나다.

ERC-20에서는 특정 단위 토큰의 소유자를 묻는 개념이 약하다.
ERC-721에서는 이 질문이 핵심이다.

### balanceOf

`balanceOf`는 특정 주소가 가진 NFT 개수를 반환한다.

```solidity
balanceOf(alice);
```

단, 이 함수는 Alice가 어떤 NFT를 가지고 있는지까지 알려주지는 않는다.
몇 개를 가지고 있는지만 알려준다.

그래서 실제 서비스에서는 이벤트 로그를 인덱싱하거나, `ERC721Enumerable` 같은 확장을 사용해 소유 NFT 목록을 관리한다.

### approve

`approve`는 특정 NFT 하나에 대한 전송 권한을 다른 주소에게 준다.

```solidity
approve(marketplace, 1001);
```

이 경우 `marketplace`는 `tokenId 1001`을 대신 전송할 수 있다.
NFT 마켓플레이스에서 판매 등록할 때 자주 사용된다.

### setApprovalForAll

`setApprovalForAll`은 특정 operator에게 내 모든 NFT 관리 권한을 준다.

```solidity
setApprovalForAll(marketplace, true);
```

사용자 입장에서는 편하지만, 보안상 위험도 크다.
악성 마켓플레이스나 피싱 사이트에 이 권한을 주면 지갑 안의 해당 컬렉션 NFT가 전부 탈취될 수 있다.

NFT 피싱에서 자주 악용되는 지점이다.

### safeTransferFrom

`safeTransferFrom`은 NFT를 전송할 때 받는 주소가 컨트랙트라면 ERC-721 수신을 지원하는지 확인한다.

그냥 `transferFrom`으로 컨트랙트 주소에 보내면 NFT가 잠길 수 있다.
받는 컨트랙트가 NFT를 처리할 방법이 없을 수 있기 때문이다.

`safeTransferFrom`은 이 문제를 줄이기 위해 받는 컨트랙트의 `onERC721Received` 응답을 확인한다.

## ERC-721 이벤트

ERC-721에서 중요한 이벤트는 세 개다.

| 이벤트 | 의미 |
| --- | --- |
| `Transfer` | NFT 소유권 이동 |
| `Approval` | 특정 tokenId에 대한 권한 승인 |
| `ApprovalForAll` | operator에게 전체 권한 승인 |

NFT 마켓플레이스와 블록 익스플로러는 이 이벤트를 추적해 소유권 이동과 승인 상태를 분석한다.

특히 `Transfer` 이벤트는 NFT 민팅, 전송, 소각 흐름을 추적하는 데 중요하다.

```text
0x000...000 -> Alice        mint
Alice -> Bob                transfer
Bob -> 0x000...000          burn
```

## 실제 사용처

ERC-721은 NFT 이미지에만 쓰이는 표준이 아니다.
고유한 소유권을 표현해야 하는 곳이라면 여러 방식으로 사용될 수 있다.

### NFT 마켓플레이스

OpenSea 같은 마켓플레이스는 ERC-721 인터페이스를 기반으로 NFT를 조회하고 거래한다.

판매자는 마켓플레이스에 `approve` 또는 `setApprovalForAll`을 준다.
구매가 발생하면 마켓플레이스 컨트랙트가 NFT를 구매자에게 전송한다.

### 게임 아이템

게임 캐릭터, 장비, 스킨, 랜드 같은 고유 자산을 ERC-721로 표현할 수 있다.

이 방식의 장점은 게임 서버 바깥에서도 소유권을 확인할 수 있다는 점이다.
단점은 모든 게임 자산을 온체인으로 관리하면 비용과 UX 문제가 커질 수 있다는 점이다.

### 멤버십과 티켓

특정 NFT를 가진 사람만 접근할 수 있는 커뮤니티, 이벤트, 콘텐츠를 만들 수 있다.

예를 들어 다음 구조가 가능하다.

```text
지갑이 특정 ERC-721을 보유하고 있는지 확인
보유 중이면 입장 허용
보유하지 않으면 접근 거부
```

이런 방식은 NFT를 단순 이미지가 아니라 접근 권한 증명으로 사용하는 예다.

### 인증서와 소유권 증명

수료증, 라이선스, 디지털 인증서 같은 고유 증명에도 사용할 수 있다.
다만 개인정보나 법적 효력 문제는 별도로 고려해야 한다.

블록체인에 올라간 토큰이 실제 세계의 권리를 자동으로 보장하는 것은 아니기 때문이다.

## ERC-20과 ERC-721 비교

ERC-721을 이해하려면 ERC-20과 비교하는 것이 가장 빠르다.

| 항목 | ERC-20 | ERC-721 |
| --- | --- | --- |
| 토큰 성격 | 대체 가능 | 대체 불가능 |
| 기본 단위 | 수량 | tokenId |
| 소유 구조 | 주소별 잔액 | tokenId별 소유자 |
| 전송 대상 | amount | tokenId |
| 대표 사용처 | 코인, 거버넌스 토큰, 포인트 | NFT, 게임 아이템, 티켓, 멤버십 |
| 메타데이터 | 표준 핵심 아님 | tokenURI 확장으로 자주 사용 |

핵심 차이는 `amount`와 `tokenId`다.

ERC-20은 몇 개를 보낼지가 중요하다.
ERC-721은 어떤 토큰을 보낼지가 중요하다.

---

## ERC-721의 한계

ERC-721도 완벽한 표준은 아니다.

### 가스 비용

NFT를 하나씩 전송하고 관리하는 구조라 대량 발행이나 대량 전송에 비효율적일 수 있다.

이 문제 때문에 ERC-1155 같은 멀티 토큰 표준이 등장했다.
ERC-1155는 대체 가능 토큰과 대체 불가능 토큰을 하나의 컨트랙트에서 함께 다룰 수 있다.

### 메타데이터 저장 문제

NFT 이미지는 보통 블록체인에 직접 저장되지 않는다.
대부분 `tokenURI`가 가리키는 외부 저장소나 IPFS에 저장된다.

그래서 다음 문제가 생길 수 있다.

```text
메타데이터 서버가 사라짐
이미지 링크가 바뀜
프로젝트가 tokenURI를 임의로 변경함
```

NFT를 분석할 때는 토큰 자체뿐 아니라 메타데이터가 어디에 저장되는지도 봐야 한다.

### 승인 권한 위험

`setApprovalForAll`은 편하지만 위험하다.
사용자가 악성 컨트랙트에 전체 권한을 주면 해당 컬렉션 NFT가 모두 위험해진다.

NFT 보안 사고에서 승인 권한 문제는 매우 자주 등장한다.
그래서 지갑과 마켓플레이스는 사용자에게 어떤 권한을 주는지 명확히 보여줘야 한다.

### 소유권과 법적 권리의 차이

ERC-721 토큰을 소유한다고 해서 항상 이미지 저작권이나 실물 자산 권리를 갖는 것은 아니다.

온체인 소유권과 오프체인 법적 권리는 다르다.
이 부분을 구분하지 않으면 NFT를 과대해석하게 된다.

---

## EIP-721의 의미

EIP-721의 핵심은 NFT라는 유행어 자체가 아니다.
고유한 디지털 자산을 표준 인터페이스로 다룰 수 있게 만든 점이 중요하다.

분석 관점에서 보면 다음 의미가 있다.

| 관점 | 의미 |
| --- | --- |
| 기술적 관점 | tokenId 기반 소유권 모델을 표준화했다 |
| 생태계 관점 | 지갑, 마켓플레이스, 게임, 인덱서가 NFT를 공통 방식으로 다룰 수 있게 했다 |
| 보안 관점 | 승인 권한과 메타데이터 신뢰 문제가 중요해졌다 |
| 경제적 관점 | 디지털 자산의 희소성과 거래 가능성을 만들었다 |

ERC-721은 단순히 NFT 이미지를 위한 표준이 아니다.
`고유한 자산을 온체인에서 어떻게 표현할 것인가`에 대한 표준이다.

---

## 정리

EIP-721은 대체 불가능한 토큰을 표현하기 위한 표준이다.
ERC-20이 수량 중심 토큰이라면, ERC-721은 `tokenId` 중심 토큰이다.

EIP-721은 다음 문제를 해결했다.

```text
각 토큰을 고유하게 식별하는 문제
특정 tokenId의 소유자를 추적하는 문제
NFT 마켓플레이스와 지갑의 호환성 문제
고유 자산을 스마트 컨트랙트에서 다루는 문제
```

하지만 한계도 있다.
가스 비용, 메타데이터 저장 위치, 승인 권한 위험, 법적 권리와 온체인 소유권의 차이를 같이 봐야 한다.

결국 EIP-721을 읽을 때 중요한 것은 함수 목록을 외우는 것이 아니다.
왜 ERC-20만으로는 부족했는지, 그리고 고유한 자산을 표현하기 위해 어떤 모델을 선택했는지를 이해하는 것이다.
